# ADR 0001 · 协议端选型：NapCatQQ

- **Status**: Accepted
- **Date**: 2026-04-28
- **Decider**: @owner
- **Supersedes**: —

## 背景

QQ 机器人需要一个"协议端"登录 QQ 并暴露标准接口（OneBot v11）供业务框架调用。
截至 2026 年，候选方案有：

| 方案 | 语言 | 协议基础 | 维护状态 |
|---|---|---|---|
| **NapCatQQ** | TS (NTQQ 内核) | OneBot v11 | 活跃 |
| Lagrange.OneBot | C# | 自研协议栈 | 活跃 |
| LLOneBot | TS (NTQQ 内核) | OneBot v11 | 活跃 |
| go-cqhttp | Go | mirai 协议 | **已停更** |
| Mirai | Java | 自研 | 收缩 |
| 官方 QQ Bot API | — | Webhook | 官方支持，但群聊受限 |

## 决策

**采用 NapCatQQ 作为协议端**，同时保留通过 adapter 切换到官方 API 的可能。

## 理由

### 为什么 NapCatQQ
1. **基于 NTQQ 内核**：跟随腾讯官方客户端版本，协议稳定性高
2. **功能完整**：群聊、私聊、图片、语音、表情、戳一戳、合并转发、群公告等全支持
3. **活跃维护**：2025-2026 年持续更新，社区问题响应快
4. **部署简单**：官方 Docker 镜像开箱即用，有 Web UI
5. **OneBot v11 协议**：生态最广，NoneBot 一等公民支持
6. **资源占用低**：~200MB 内存，远低于 Mirai

### 为什么不选其他

- **Lagrange.OneBot**：功能稍弱（部分高级消息类型不支持），NTQQ 版升级依赖开发者手动跟进
- **LLOneBot**：功能与 NapCat 重叠但社区更小
- **go-cqhttp**：已停更，协议失效风险高
- **Mirai**：Java 依赖重、社区收缩
- **官方 QQ Bot API**：
  - 群聊权限审核严格
  - 富媒体与互动受限（Markdown/Ark 模板需申请）
  - 不适合"多功能小游戏"这种重交互场景
  - **但会作为未来路径保留**，见下

## 合规性与风险

- NapCat 属于第三方协议端，存在**账号被封风险**
- 风险缓解：
  - 使用**专用小号**（不绑定主号、不绑定手机绑关系链）
  - 养号 2-3 天再接入机器人
  - 控制发送频率（Core 层内置节流）
  - 不发送敏感内容、不做营销
- 一旦封号：切换新小号，业务数据不丢（数据绑群/玩家，不绑机器人账号）

## 未来迁移路径（官方 API）

如果将来需要合规化（商业化、对外发布），迁移成本：

- Core 层用 NoneBot2 抽象，已屏蔽大部分协议差异
- 切换 adapter：`nonebot-adapter-onebot` → `nonebot-adapter-qq`
- 需要适配的点：
  - 消息类型：部分富媒体需换成官方支持的 Markdown/Ark 模板
  - 权限申请：主动消息额度、群聊权限
- 游戏层代码预计**改动 <10%**（只要不依赖非官方特性）

Core 层设计时**主动规避**仅 NapCat 独有的能力，以保留切换选项。

## 影响

- 所有部署文档以 NapCat Docker 镜像为准
- 需要一个专用 QQ 小号
- 代码里通过环境变量 `ONEBOT_WS_URL` 指向 NapCat，切换成本低
