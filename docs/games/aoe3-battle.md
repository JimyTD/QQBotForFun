## 一、玩法概述

### 模式 A：押注模式（首期实现）

1. 玩家 `@机器人 斗蛐蛐` 开一局
2. 系统在等资源预算下随机生成**红蓝两方阵容**，公布对阵面板
3. **押注阶段**，群友发押注口令下注
   - **统一口令**：`押注1`（押红方）/ `押注2`（押蓝方）
   - 数字口令的好处：避免颜色识别歧义、容错率高、便于扩展（未来如做多方混战可直接 押注3 / 押注4）
   - **面板视觉**仍用 🔴红 / 🔵蓝（颜色直观），但口令只接受 1/2，不再兼容
   - **不设倒计时**：群友可随时押注，任意群友发 `开战` 命令即可跳过等待直接开打
   - **锁死第一笔**：同一玩家押注后不可改押（发了 `押注1` 再发 `押注2` 无效）
4. 进入战斗阶段后**不再接受押注**
6. 系统**一次性完整模拟战斗**，得到完整事件流，再**按时间分批播报**
7. 结算：赢家平分输家的金币池,所有参与者获得 1 金币参与奖
8. **不自动开下一局**（与项目内其他游戏一致，由玩家手动发起）

### 模式 B：单挑模式（首期一起做）

1. 玩家 `@机器人 斗蛐蛐 单挑` 开一局
2. 系统随机抽两个兵种（包含无费用兵种），**真正的 1v1**：每方**只有 1 个单位**
3. 同样可押注，规则同模式 A

### 模式 C：黑名单乱斗（已实现）

1. 玩家 `@机器人 斗蛐蛐 乱斗`（或 `黑名单` / `黑名单乱斗`）开一局
2. 系统从 `BATTLE_BLACKLIST`（彩蛋 / 作弊码 / 怪物级战役兵 ~18 个）随机抽阵容
3. **完全无视 cost，按战力分平衡数量**——详见 §2.4
4. 押注 / 战斗 / 结算流程完全复用模式 A
5. **节目效果优先于公平**：怪兽 vs 民兵就是悬殊，独苗 vs 几只小弟也照样看戏

### 模式 D：王中王（已实现）

> 对外名称 **王中王**；内部 `GameMode.id = "rival"`。  
> 从固定 **10 个职能主题** 里抽兵对决，主题用 tag 并集/交集筛池（详见 §2.5）。

#### 入口 A：表情选主题（主路径）

1. 玩家 `@机器人 斗蛐蛐 王中王`（别名 `宿敌` / `宿敌挑战` 可选）
2. **尚未开局**——进入「选主题」阶段，**不占用** `GameRunner` slot
3. 系统从 10 个主题中 **随机抽 3 个**，发一条选单消息
4. 为该消息挂载 **3 个表情**（NapCat 消息表情回应）；群友 **点击表情** 即选定对应主题
5. **任一群友点选即生效**（第一个有效表情回应锁定主题），随后 **正式 `create_and_start` 开局**
6. **不设倒计时**：群友可随时点选；`@结束` 可取消 pending
7. **文字兜底**：对同一条选单消息回复 `1` / `2` / `3` 等价于点第 1/2/3 个表情（CLI 同样用 1/2/3，见 `docs/13-cli-bot-parity.md`）

#### 入口 B：指定主题（快捷）

1. 玩家 `@机器人 斗蛐蛐 王中王 散兵`（或 `火枪`、`近战骑` 等主题别名）
2. **跳过选主题**，直接用该主题开局
3. 可选末尾预算数字：`斗蛐蛐 王中王 散兵 15000`（范围 1000~50000，默认 10000）

#### 对阵与后续流程

- 红蓝双方各 **随机 1 个兵种**（均从该主题池抽取），数量用 **LCM 平衡资源**——**与自选模式（2 个指定兵）逻辑一致**，不是单挑 1v1
- 押注 / 开战 / 模拟 / 结算 **完全复用模式 A**
- VS 面板标题示例：`⚔️ 帝国3斗蛐蛐 · 王中王 · 散兵王`

### 模式 E：自选（已实现，简述）

- `@机器人 斗蛐蛐 自选 …` / `@机器人 斗蛐蛐自选 …`
- 玩家指定 1~2 个兵种名，LCM 平衡；详见 `lineup.py :: generate_custom_lineup`

### 模式 F：王中王锦标赛（设计封板，编码阶段）

> 对外名称 **王中王锦标赛**；内部 `GameMode.id = "rival_tournament"`。  
> 从同主题池抽 **8 种兵**，单败淘汰 + 败者组排位，完整排出 1~8 名。

#### 指令

```
@机器人 斗蛐蛐 王中王锦标赛
```

- **主题选择流程与普通王中王一致**：开局后先随机 3 主题、表情/文字选主题
- 主题同王中王（散兵/火枪/近战重步/弓手/掷弹/近战骑/远程骑/炮/亡命徒/佣兵）
- 不支持自定义预算和时代（锁定群配置）
- 别名：`锦标赛` / `tournament`

#### 赛制

单败淘汰 + 败者组排位，共 12 场比赛：

```
QF1/2/3/4（4 场）→ LR1/LR2（2 场排位）→ 7/8名战 + 5/6名战（2 场）
→ SF1/SF2（2 场半决赛）→ 季军战 → 决赛
```

#### 阶段流程（4 阶段出图 + 最终排名）

| 阶段 | 内容 | 出图 | 交互 |
|------|------|------|------|
| ① 抽签仪式 | 抽 8 兵 → 随机配对 → 赛前对阵图（全员亮色候场） | ✅ 赛前图 | 夺冠押注开放（发序号押注） |
| ② 八强战 | QF1~QF4 逐场串行播报 | ✅ 八强结果图 | 群友发 `开战` 进入下一轮 |
| ③ 败者组排位 | LR1→LR2→7/8名战→5/6名战 逐场串行 | ❌ 不出图 | 同上 |
| ④ 半决赛+季军战 | SF1→SF2→季军战 逐场串行 | ✅ 半决赛结果图 | 同上 |
| ⑤ 决赛+排名榜 | 决赛播报 → 冠军图 → 最终排名图 → 押注结算 | ✅ 决赛图 + 排名图 | 自动结束 |

> **败者组排位赛不出对阵图**，仅文字战报播出，排位结果反映在最终排名图中。

#### 对阵图方案：Pillow 合成图（确定）

使用 Pillow 生成 940×660 的括号淘汰树图片，包含：
- 8 个兵种图标 + 中文名
- 淘汰连线（晋级/淘汰/待定三种颜色）
- 轮次标题渐进高亮（八强冷灰蓝→四强亮银→决赛暖金→冠军纯金）
- 胜者浅金色名称、败者暗灰 + 红叉
- 候场状态用亮色图标（不暗化）
- 右下角操作提示文本（如「发送「开战」开始八强战」）

4 个阶段的图各自状态：
- **赛前图**：全员亮色 + 待定虚线 + `?` 占位 + 押注提示
- **八强结果图**：八强胜败已分，四强晋级者亮出，后续待定
- **半决赛结果图**：八强+四强已填，决赛选手亮出，冠军待定
- **决赛结果图**：完整图 + 金色冠军路径 + 冠军大框

最终排名图为独立的 400×540 排名列表图片。

#### 押注机制

- 💰 **赛前夺冠押注**：抽签后开放，群友发**序号**（1~8）押对应兵种夺冠
- 赛前图的押注说明中列出兵种序号对应关系
- ❌ 不做每场/每轮独立押注
- 锁死第一笔，不可更改
- 决赛后按夺冠结果结算

#### 设计决策

> 以下决策来自设计讨论（2026-06-16），固化于此。

**1. 兵种数据展示**
- ❌ **不展示任何兵种属性数值**（血量/攻击/ROF/射程/护甲等）
- ❌ **不展示兵种信息卡**
- 理由：简略数值（攻击/ROF/护甲）在缺少防御/倍率/射程等上下文时会产生严重误导
- 替代方案：括号图 + 图标供辨识，群友可通过 `@机器人 查aoe3 兵种名` 自行查询

**2. 开局对阵图**
- ✅ **Pillow 合成图方案**：经原型验证可行，输出效果佳
- 4 阶段出图时机见上方阶段流程表
- 败者组排位赛不出对阵图，仅文字战报，排名体现在最终排名图

**3. 对阵配对**
- 🔀 **完全随机配对**，不做 seeding
- 禁用心智模型中的 `power_score` 排序（该公式为黑名单乱斗配兵专用，不计倍率/射程/速度，不适用于排名预测）

**4. 科技规则**
- 🔒 **强制关闭 random generic techs**（无论群配置如何）
- ✅ **遵守群设定的时代和时代科技逻辑**（包括 `age`、兵种升级 `upgrades`）
- 锦标赛开局时 **锁定时代快照**，群友中途修改群时代配置**不影响已开局的锦标赛**

**5. 群资源占用**
- 🚫 **锦标赛进行中，该群不能开其他任何 AOE3 斗蛐蛐局**
- 同一群一次只允许一个锦标赛（复用现有的 `GameRunner` 一群一局机制）

**6. 阶段推进节奏**
- ⏭ **轮内自动串行**（同一轮内的 QF1→QF2→QF3→QF4 等逐场自动播报）
- ⏸ **轮间等"开战"**：一轮全部打完后，更新对阵图 + 右下角提示下一轮，群友发一次 `开战` 推进一轮
- ❌ 不设自动倒计时

**7. 称号体系**
- 暂时与普通王中王保持一致（如「散兵王」「火枪王」等）

**8. 不包含的字段/UI**
- ❌ 不输出普通斗蛐蛐中的"开战话术"（`⚔️ 战斗打响！…`）
- ❌ 不输出全灭播报（`xxx 全灭` 台词）
- ❌ 不输出详细战斗动态（死亡动词等）
- ✅ 保留血条、存活数、击杀数、伤害值的战斗报告

## 二、阵容生成

### 2.1 资源预算（押注模式）

- 基础预算：**10000 资源**（默认，可通过 `@机器人 斗蛐蛐 2000` 自定义，范围 1000~50000）
- 食物、木材、金币**价值一致**，单位总资源 = food + wood + gold
- 具体数量计算见 §2.2.1（v1 单兵种）和 §2.2.3（v2 复合阵容）

### 2.2 阵容构成（押注模式）

#### 2.2.1 v1 单一兵种（已实现）

- 红蓝双方各**随机抽 1 个兵种**
- 数量计算采用**近似 LCM 算法**（详见 §2.2.4），让双方资源尽量相等
- 示例：红方 12 火枪手（cost 100×12=1200）vs 蓝方 2 鹰炮（cost 600×2=1200）
- 双方总资源**尽量靠近**即可（不做强制重抽）

#### 2.2.2 兵种池（押注模式）

- 基础池：有费用（`cost > 0`）且有攻击力（`has_attack=True`）且有 HP（`hp > 0`）的兵种
- **保留：雇佣兵**（群友最爱）
- **保留：英雄、宠物**（不再按类型排除，增加趣味性）
- 排除：建筑马车、**船只**（type 含 `Ship` 的单位；船只数据质量差且一维场地上表现极端）
- **全局排除**：`is_excluded_unit()`（`repository.py`）—— 召唤占位符（`*batch` / `*armyspawn`）、
  PVE 守护者（type 含 `Guardian`）、八旗军/领事馆/原住民代币（type 含 `AbstractBannerArmy`）。
  这些不是真单位，搜索和对战池都拦。
- **黑名单排除**：`lineup.py` 中的两组集合（两种模式统一生效）
  - `BLACKLIST`：数据 broken / 无法模拟，**永久禁用**（火船 ROF=0、假炮、作弊单位等）
  - `BATTLE_BLACKLIST`：彩蛋 / 作弊码 / 剧情兵（镭射熊、垮盛顿、怪兽卡车、加特林骆驼、帝国护卫舰…），
    **仅普通对战禁用，搜索仍可见**。预留给计划中的"黑名单乱斗"模式（从这个池子里抽兵互殴）。

#### 2.2.3 v2 复合阵容（已实现）

每方可出 **1~3 个兵种**，系统随机抽取 + 自动分配数量。红蓝双方**独立生成**（兵种数可以不同）。

##### 兵种数量权重

```python
slot_count = rng.choices([1, 2, 3], weights=[45, 35, 20], k=1)[0]
# 1 种兵 45%（最常见，保持经典单兵种对决的频率）
# 2 种兵 35%（主力+辅助的经典搭配）
# 3 种兵 20%（偶尔出现的豪华阵容）
```

红蓝双方各自独立抽取兵种数，互不影响（可能出现红方 1 种 vs 蓝方 3 种）。

##### 抽兵约束

抽到的 N 个兵种必须满足：**各出 1 个的总 cost ≤ 该方预算**，否则重抽。

```python
# 例：预算 1000，抽到 鹰炮(600) + 铁甲舰(800) + 护卫舰(1000)
# 保底消耗 = 600 + 800 + 1000 = 2400 > 1000 → 重抽
```

这保证了后续贪心填充算法的"每种保底 1 个"前提安全。

##### 多兵种资源分配：贪心填充三步法

```
算法：GreedyFill(budget, unit_costs[])
──────────────────────────────────────
输入：总预算 budget，兵种 cost 列表 unit_costs（已通过抽兵约束）
输出：每个兵种的数量 counts[]

Step 1 — 保底：每种兵各 1 个
  counts = [1, 1, ..., 1]
  remaining = budget - sum(unit_costs)

Step 2 — 均分：剩余预算均分给每个兵种
  per_budget = remaining // N
  对每个兵种 i：
    extra = per_budget // unit_costs[i]
    counts[i] += extra

Step 3 — 贪心零头：最后的零头逐个加最便宜的兵
  remaining = budget - sum(counts[i] * unit_costs[i])
  while remaining > 0：
    找 cost ≤ remaining 的最便宜兵种，+1
```

**设计要点**：
- Step 2 的均分保证贵兵种也能分到合理数量（不会全被便宜兵吃掉）
- Step 3 的贪心只处理零头（通常 < 最便宜兵种的 cost）
- 无人口上限，数量完全由预算决定

##### 算法验证用例

| 阵容 | 预算 | 结果 | 花费 | 浪费 |
|------|------|------|------|------|
| 火枪(100)+鹰炮(600) | 1000 | 火枪×4 + 鹰炮×1 | 1000 | 0 |
| 黑骑士(500)+火枪(100) | 1000 | 黑骑士×1 + 火枪×5 | 1000 | 0 |
| 黑骑士(500)+祖瓦夫(300)+火枪(100) | 3000 | 黑骑士×2 + 祖瓦夫×3 + 火枪×11 | 3000 | 0 |
| 泽纳塔(250)+火枪(100) | 2000 | 泽纳塔×4 + 火枪×10 | 2000 | 0 |
| 弩手(40)+长矛(75)+散兵(90) | 1000 | 弩手×10 + 长矛×4 + 散兵×3 | 970 | 30 |
| 铁甲舰(800)+火枪(100)+散兵(90) | 3000 | 铁甲舰×1 + 火枪×7 + 散兵×16 | 2940 | 60 |

#### 2.2.4 近似 LCM 算法（单兵种资源平衡）

双方都是 1 种兵时，使用近似 LCM 让双方资源尽量相等：

```
算法：ApproxLCM(cost_a, cost_b, base_budget)
──────────────────────────────────────────────
1. ca, cb = round(cost_a), round(cost_b)
2. lcm = LCM(ca, cb)
3. 如果 lcm > base_budget × 1.3：
     → LCM 太大，退化为 base_budget
4. 否则：
     actual = round(base_budget / lcm) × lcm
     actual = clamp(actual, base_budget × 0.7, base_budget × 1.3)
5. red_count = actual // ca
   blue_count = actual // cb
```

**预算浮动范围 ±30%**（不会偏离太远）。

#### 战力分公式（V2 几何平均 + 双层溢出减益）

```python
ARMOR_WEIGHT          = 0.3   # 护甲轻微增益（armor=0.9 仅 ×1.27）
BLACKLIST_AOE_DPS_MULT = 0.3   # AOE 单击伤害加成（每 1 半径 +30%）
HIT_BASELINE          = 50    # 单击伤害基准
DPS_BASELINE          = 20    # DPS 基准

def _soft_diminish(x, base):
    """x ≤ base 时线性；x > base 时按 sqrt 收益（边际递减）。"""
    return x if x <= base else base + sqrt((x - base) * base)

def power_score(unit):
    # HP：线性单调递增 + 护甲轻微增益
    hp_eff = hp × (1 + max(armor_r, armor_m) × ARMOR_WEIGHT)

    # 单击伤害（含 AOE 加成）先溢出减益（治"800 攻击打民兵"的浪费）
    hit_r = _soft_diminish(atk_r × (1 + aoe_r × 0.3), HIT_BASELINE)
    hit_m = _soft_diminish(atk_m × (1 + aoe_m × 0.3), HIT_BASELINE)

    # 除以 ROF 得 DPS，再做一次溢出减益（治"机关枪打蚊子"的浪费）
    dps_eff = _soft_diminish(max(hit_r/rof_r, hit_m/rof_m), DPS_BASELINE)

    # 几何平均：HP/DPS 同等加权，单项极端高不会让总战力线性爆炸
    return sqrt(hp_eff × dps_eff)
```

设计三要素（与玩家直觉对齐）：

1. **HP**：单调递增（恒成立），sqrt 让大血量不线性主宰；护甲只做轻微增益
   （旧公式 `1/(1-armor)` 在 armor=0.9 时放大 10 倍，纯虚高）。
2. **DPS**：由 atk/rof 计算，**双层边际递减**——单击 800 攻击对一个 hp 500
   的小兵浪费 300（高射炮打蚊子），公式按 sqrt 收敛而非线性贡献。
3. **几何平均**：HP 与 DPS 同等重要，单独极端值不会线性放大。

**只看模拟器实际使用的 `attack_ranged` / `attack_melee` 槽位**：

- `attack_siege` 是 BuildingAttack 系列（§3.9），模拟器不读，不计入。
- **倍率（multipliers）不计**——克制天然在战斗里发挥作用，加进战力分会高估。
- 不计 speed / 射程——1v1 静态战力公式无法量化"接战阶段"影响。

#### 实测标定（V2 公式 vs 模拟器实战）

| 1 镭射熊 = ? | 公式预测 | 模拟器实测均衡点 | 偏差 |
|---|---|---|---|
| 重型火炮(无人口) | 19:1 | 15:1 | 1.2× |
| 防卫据点加农炮 | 12:1 | 7:1 | 1.8× |
| 民兵长 | 55:1 | 30:1 | 1.8× |

跨档偏差从旧公式的 **几十~几千倍** 压到 **1~2 倍**。剩余偏差来自剪刀石头布
（armor 实际抵消、AOE 群体放大、射程接战节奏等），单兵静态公式无法覆盖。

#### 数量分配：共用 anchor `max(双方 max) × r`

```python
SLOT_WEIGHTS_BLACKLIST = [50, 50, 0]    # 1/2/3 种兵的概率（黑名单池小，去掉 3 兵种）
BLACKLIST_R_MIN, BLACKLIST_R_MAX = 3.0, 6.0

# 抽兵（双方独立，无 cost 约束，不限制兵种组合）
red_units  = rng.sample(pool, rng.choices([1,2,3], SLOT_WEIGHTS_BLACKLIST))
blue_units = rng.sample(pool, rng.choices([1,2,3], SLOT_WEIGHTS_BLACKLIST))

# 共用 anchor：让两边都向同一个 target 配兵，总战力自然对齐
anchor = max(max(red_scores), max(blue_scores))
r = rng.uniform(R_MIN, R_MAX)
target = anchor × r

red_counts  = _fill_to_target(target, red_scores)
blue_counts = _fill_to_target(target, blue_scores)
```

`_fill_to_target` 策略：每种保底 1 个，然后反复挑 `counts[i] × scores[i]` 最小
的兵种 +1，直到总分 ≥ target × 1.05。"挑当前总分最小的 +1" 实现多兵种均匀
ramp。**无个体数上限**——与押注/单挑模式一致（§3.2），数量完全由战力分决定。

**关键决策**：

1. **共用 anchor**（旧版用本方 min × r）：旧策略下一方有怪兽就保底 1 只、
   弱方按本方 min ramp，跨方战力悬殊到 20~37×。共用 anchor 让两边都向同
   一个 target 配兵，跨方战力比压到 1~4×。
2. **无数量上限**：与押注/单挑同哲学——1 镭射熊 vs 500 民兵就是黑名单乱斗
   的常规节目效果。"超人打一堆蚂蚁"也是合法节目。模拟器在 §2.2.5 的压测
   已验证 500 个体 < 1 秒可控。
3. **不约束兵种组合**：保留"加藤大名 vs 镭射熊"这种荒诞局的可能性——克制和
   AOE 由模拟器自然发挥，公式不做事前过滤。

## 三、战斗模拟

### 3.0 模拟与播报的关系（重要架构决策）

**采用"先全量模拟、再分批播报"模式**：

```
押注结束 → [一次性完整模拟战斗，生成完整事件流] → [按时间窗分批吐日志]
```

- **优点**：模拟过程不受播报延迟干扰；播报失败可重试；测试可绕过播报直接看结果
- **播报节奏**：每 **2 秒模拟时间**（即 20 tick）汇总成一条消息发出
- **零伤亡片段**：该 2 秒窗口内**双方均无死亡**则**跳过**（不发消息也不 sleep，直接进入下一窗口），避免冷场刷屏
- **关键节点**：开战、首杀、半灭线、最后一击、战斗结束 单独成条优先播报
- **真实节奏**：有伤亡的窗口播报后 sleep 2 秒，让群友有时间看，节奏接近真实战斗

### 3.1 场地模型

- **一维直线场地**，中间空地长度 30 距离单位
- 双方在空地两侧**按排列阵型展开**：
  - 红方从 `pos=0` 往负方向排布，蓝方从 `pos=30` 往正方向排布
  - 近战在前排（靠近空地），远程在后排
  - 每排站 `ROW_CAPACITY`（默认 8）人，排间距 `ROW_SPACING`（默认 1.5）
  - 同一排的所有士兵 pos 完全相同 → 对面集火一排而非一个人
  - 阵地最大纵深 `FORMATION_MAX_DEPTH`（默认 12），排太多时自动压缩排间距
- 距离 = |士兵A位置 - 士兵B位置|

### 3.2 士兵个体

- 每个士兵有**独立 HP**（如 5 个火枪手 = 5 个各 150HP 的个体）
- 每个士兵有**独立攻击 CD 计时**
- 每个士兵有**锁定目标**（一个具体的敌方个体）
- 无人口上限，个体数完全由资源预算和兵种 cost 决定

### 3.3 时间推进

- 模拟以 **tick** 为单位推进（tick 间隔 0.1 秒）
- 每 tick 依次处理：移动 → 目标锁定 → **同时开火结算** → 死亡清算 → 事件入队
- 设置**最大模拟时长**（暂定 60 秒 = 600 tick），超时按**剩余单位总资源价值**判胜（存活单位数 × 单位 cost）；单挑模式超时则判**平局**
- 模拟器输出**结构化事件流**（ATTACK / DEATH / TARGET_LOCK / AOE_SPLASH / BATTLE_END），话术层从事件流生成播报文本

#### 同时开火结算规则（核心，所有兵种通用）

每 tick 的攻击阶段采用**同时开火**模型，而非串行结算。此规则对远程、近战、炮兵一视同仁：

1. **收集攻击者**：所有 CD 好且已锁定目标的士兵，在本轮"出手"——攻击已发出，不可撤回（远程=弹丸出膛，近战=刀已挥出）
2. **进入 CD**：所有攻击者立即进入攻击冷却（ROF），不管后续结算结果
3. **随机顺序结算**：将所有攻击者（跨阵营）随机打乱顺序，依次结算伤害：
   - 命中时目标**还活着** → 正常扣血 + 触发 AOE 溅射
   - 命中时目标**已死**（被本轮其他人先打死） → 伤害浪费，**不触发 AOE**
4. **死了也能出手**：即使某士兵在本轮结算中被别人杀死，它的攻击依然命中（步骤 1 已标记）

**设计意图**：
- **火力浪费**：多人锁同一目标时产生 overkill，远程兵/炮兵天然更易浪费火力（射程远 → 更多人锁同一个前排目标），近战兵也会（多人围殴同一个敌人，第一刀砍死了后面的刀砍空）
- **消除先手优势**：不再因为 ID 小而先开火、先杀人，双方对等
- **AOE 不炸尸体**：命中活目标才触发溅射，打尸体不爆——避免 AOE 兵种获得不真实的溅射覆盖

### 3.4 移动规则

- 每 tick 每个士兵个体向最近敌方移动 `速度 × tick_interval` 距离
  - **speed 字段单位 = 每秒移动距离**（即 `speed=4.0` 表示每秒移动 4 个距离单位）
  - 同方同兵种个体位置重叠（无碰撞体积），实际效果等同于整组移动
- **F2A 逻辑：当士兵能对任意敌方发起攻击时，立即停止移动**
- 具体规则：
  - 有远程/攻城攻击的兵种：与最近敌方距离 ≤ 射程 → **停止移动，开火**
  - 纯近战兵种：与最近敌方距离 ≤ 近战射程（1.5）→ **停止移动，近战**
  - 已停止的远程兵被敌近战兵贴到 < range_min：**不主动后退**，切换到近战或缩减伤害模式（见 §3.5）

### 3.5 攻击判定

#### 攻击模式选择（优先级，逐个目标判定）

对于同时有远程和近战的兵种，根据与锁定目标的距离：

1. **距离 ≤ 1.5**（近战射程）→ 使用**近战攻击**
2. **1.5 < 距离 ≤ range 且 距离 ≥ range_min** → 使用**远程攻击**
3. **距离 < range_min 且 > 1.5**：
   - 若有近战攻击力 → **等待敌方继续靠近到距离 ≤ 1.5**，然后近战（自身不后退也不前进，因为已因 F2A 停下）
   - 若无近战攻击力 → **原地缩减伤害远程攻击**（伤害 × 50%，详见下方"贴脸惩罚"）
4. **距离 > range** → 继续移动靠近

> 注：当 range_min ≤ 1.5 时，情况 3 不会发生（从远程模式直接切到近战模式）

#### 纯远程兵种（无近战攻击力）

- 进入射程后停止移动、开火
- 被近身（距离 < range_min）→ 缩减伤害远程攻击（50%）
- 不会主动前进

#### 贴脸惩罚（"远程衰减 50%"的真实含义）

- **触发条件**：射击距离 < range_min（最小射程）**且**该兵种**无近战攻击力**
- **效果**：原地继续远程攻击，但**伤害 × 50%**
- **意义**：这是**简化的贴脸惩罚建模**，不是 AoE3 真实游戏机制
  - AoE3 真实情况下，远程兵被贴脸会触发寻路/逃离/换近战等复合 AI 行为
  - 这些复合行为在简化模拟里难以 1:1 还原，因此用"原地继续打但伤害减半"代替
- **不是**：距离衰减曲线 / miss 率 / 装填中断（这些机制本游戏均**不实现**）
- **数据来源**：50% 是建模参数，AoE3 官方未公布对应数据。若未来找到真实数据可调整

#### 攻击抬手（Windup）

一次完整的攻击动画可拆分为三个阶段：
- 远程攻击：攻击前摇 → 投掷物发射 → 攻击后摇
- 近战攻击：攻击前摇 → 攻击命中 → 攻击后摇

**ROF = 前摇 + 命中/发射 + 后摇**（三段加起来就是面板显示的攻击频率）。

**Windup（抬手）不包含在 ROF 里**——它是"从静止状态进入攻击循环"的额外延迟：

```
移动中 → 射程内有敌人 → 停下 → attack_cd = windup → 等待抬手 → 开火
→ CD = ROF → 开火 → CD = ROF → ...（连续攻击不重复抬手）
```

**核心规则**：
1. **停下 = 开始抬手**：士兵 `stopped` 从 False 变为 True 的那一刻，根据初始攻击模式设 `attack_cd = windup_ranged` 或 `windup_melee`
2. **连续攻击不重复 windup**：站着打就是纯 ROF 循环
3. **中断攻击（移动）后再攻击要重新 windup**：在线性对冲模型中，中断攻击 = 移动（射程内没人了 → 继续前进 → 再次停下时重新抬手）
4. **转火不重新 windup**：目标死亡后重新锁定新目标，只要 `stopped` 没变就不重新抬手
5. **抬手期间目标死了**：windup CD 归零后正常走锁定→开火。如果射程内还有活目标则转火命中；如果射程内无人则火力浪费（这是长弓手等慢抬手单位的天然劣势）

**windup 值确定时机**：在停下的那一刻就能确定——因为 `_can_attack_any_enemy` 已经知道是远程射程够到了还是近战射程够到了。

**数据来源**：`units.json` 中的 `windup_ranged` / `windup_melee` 字段，由 parser 从 anim XML 的 `<tag type="Attack">` 解析。

**面板展示**：查兵种面板展示 windup 数据，斗蛐蛐战前面板和播报不展示。

##### 已知精度问题（2026-06 调查，暂不修复）

以下两个问题经调查确认存在，但作为群小游戏精度足够，暂搁置。

**问题 1：windup 数值可能不精确**

当前 parser 直接将 anim XML 的 `<tag type="Attack">` 值当作秒数。但 Steam modding 指南
（[Proto & Anim Action & Logic Lists](https://steamcommunity.com/sharedfiles/filedetails/?id=821258879)）
明确写道 tag 值是**动画运行时间的百分比位置**（0.0~1.0），Fandom Wiki Attack Delay 词条也在
Bang Engine 章节确认 *"attack delays are handled with animation completion percentages in tandem
with tags"*。正确公式应为 `windup_seconds = tag_fraction × GrannyAnim固有时长`，需从 .gr2
二进制文件提取动画时长。

但 ESOC 社区和 Fandom Wiki 各单位页面又都直接把 tag 值报成秒数（如长弓 0.98s、散兵 0.46s），
可能是多数攻击动画时长接近 1 秒所以误差不大。有资深玩家提供的数据（长弓 1.47s、弩手 0.35s）
与 tag × 动画时长模型吻合，但未能用第三个单位做决定性验证。

修复方案：写 .gr2 时长提取器（参考 [AOE3GR2Exporter](https://github.com/zzqqqzzz/AOE3GR2Exporter)），
乘以 tag 百分比得到真实秒数。工作量中等，收益有限。

**问题 2：转火是否应重新抬手**

当前模拟器在目标死亡后无缝转火（`_acquire_target` 只换 `target_id`，不重置 `attack_cd`），
即转火不重新 windup。但 AoE3 DE 玩家观察到目标死亡后单位行为异常（如炮兵冲向死去目标的位置），
暗示引擎在目标死亡时**会中断攻击周期**。未找到 AoE3 社区或 modding 文档对"站定单位 auto-retarget
后是否重新 windup"的明确结论。若实际会重新 windup，则慢抬手兵种（长弓等）的 overkill 代价
会被我们低估。

#### ROF（攻击间隔）

- ROF 直接使用 `units.json` 中的真实数据（`rof_ranged` / `rof_melee` / `rof_siege`）
  - **rof_* 字段单位 = 秒（攻击间隔）**，不是攻击频率（RPS/RPM）。即 `rof_ranged=3.0` 表示每 3 秒攻击一次
- 攻击当帧结算（不模拟弹道飞行时间），伤害立即生效

#### 缺失 ROF 的处理

- 远程 ROF 缺失 → 默认 3.0 秒
- 近战 ROF 缺失 → 默认 1.5 秒

#### 不实现的机制（明确排除）

为保持模拟简洁，**以下机制不予实现**：
- ❌ miss / 闪避率
- ❌ 距离衰减曲线（射程内伤害随距离递减）
- ❌ 装填中断 / 重新瞄准时间
- ❌ 弹道飞行时间
- ❌ 散射 / 抛物线落点
- ❌ 兵种特殊技能（霰弹模式、践踏模式、英雄主动技等）

### 3.6 目标选择

- 士兵**停止移动后立即锁定目标**（不是预先锁定）
- **锁定优先级：射程内最近的存活敌方个体**
  - 射程内只有一个敌方 → 直接锁定
  - 射程内有多个敌方且**距离不同** → 锁定**最近**的
  - 射程内有多个敌方且**距离相同**（如同兵种重叠） → **随机**选一个锁定
- **锁定目标直到其死亡**，当次攻击伤害溢出不转火
- 目标死亡后**下一个 tick 重新按上述优先级锁定新目标**（看射程内谁最近，距离相同才随机）
- **攻击阶段不重新锁定**：目标锁定在每 tick 攻击之前统一完成，开火阶段中即使目标被别人打死也不会临时换目标（打尸体 = 火力浪费）

> **设计意图：排布阵型 + F2A 对冲 + 火力浪费**  
> 双方按排列展开（近战前排、远程后排），"最近优先"锁敌自然集火前排。  
> 前排死光后，下一排距离最近被锁——逐排清扫，避免火力分散零伤亡。  
> 攻击阶段不换目标保证了真实的火力浪费：多门炮锁同一个兵，第一门杀了它，其余全是打尸体。  
> 参数可调：`ROW_CAPACITY`（每排人数）、`ROW_SPACING`（排间距）、`FORMATION_MAX_DEPTH`（最大纵深）。
### 3.7 伤害计算

#### 公式

```
伤害 = 基础攻击力 × 倍率乘积 × (1 - 目标对应抗性)
伤害最低为 1
```

#### 三种伤害类型与三种抗性的对应关系

AOE3 有三种伤害类型，每种只被对应的抗性减免：

| 伤害类型 | 对应抗性 | 数据字段 |
|---------|---------|---------|
| 远程伤害 (Ranged) | 远程抗性 | `armor_ranged` |
| 近战伤害 (Hand/Melee) | 近战抗性 | `armor_melee` |
| 攻城伤害 (Siege) | 攻城抗性 | `armor_siege`（极少见，数据中暂无） |

**关键含义**：
- 鹰炮的攻城攻击打步兵 → 步兵通常没有攻城抗性 → 0%减免 → 全额伤害
- 这就是为什么炮兵打无甲单位特别疼
- 绝大多数兵种只有近战抗性或远程抗性（或两者之一），极少有攻城抗性
- 数据中同时有双抗的兵种只有 5 个（如阿扎普、利菲迪骑士等）

#### 数据缺失：伤害类型标记

当前数据**没有标记每种攻击的伤害类型**。大多数情况下：
- `attack_ranged` → 远程伤害（吃 `armor_ranged`）
- `attack_melee` → 近战伤害（吃 `armor_melee`）
- `attack_siege` → 攻城伤害（吃 `armor_siege`，几乎无人有）

**特殊情况**（如 Abus Gunner 的远程攻击实际是攻城伤害类型）暂按上述默认规则处理。
后续如需精确可爬取 `damage_type` 字段。

#### 倍率计算

- 目标可能同时属于多个类型（如 [Cavalry, Shock infantry]）
- 攻击者的倍率列表中，所有匹配目标类型的**倍率叠乘**
- 例：攻击者对 Cavalry ×0.5、对 Shock infantry ×2.0 → 最终倍率 = 0.5 × 2.0 = 1.0
- 无匹配倍率时 → 倍率 = 1.0

### 3.8 AOE（范围伤害）

#### 设计原则

- **不考虑兵种的特殊攻击模式**（如炮兵霰弹模式、骑兵践踏模式等），每个兵种只使用其默认/主攻击模式
- 场地保持一维线性设计（评估结论：二维平面对文字播报无额外体验收益，复杂度不值得）

#### 数据来源

- 直接从游戏文件 `protoy.xml` 的 `<protoaction>` → `<damagearea>` 字段解析（`scripts/crawler/aoe3_gamedata_parser.py`）
- 当前 `units.json` 中 **147 个单位**带 AOE 数据（重型加农炮 / 迈索尔火箭炮 aoe=4，胸甲骑兵 melee aoe=2，战舰类 aoe=1 等）
- 每个攻击模式的 AOE 与该模式的 damage / range / multipliers 整体绑定，不存在拼接

#### AOE3 原版溅射机制（解包 + 社区实测，引擎逻辑不在 XML 里）

**解包可见字段**（`protoy.xml` → `<protoaction>`，与代表攻击整包绑定）：

| 字段 | 含义 |
|------|------|
| `<damagearea>` | 溅射半径（游戏内 AoE 圆半径） |
| `<damagecap>` | **溅射总伤害池**（不含主目标全额伤害；主目标另算） |
| `<basedamagecap>` | 多为 `1`，表示伤害上限随科技/升级按 BasePercent 缩放（见 `techtreey.xml` 的 `subtype="DamageCap"`） |

**解包里没有**溅射分配公式（距离衰减、随机选目标、Bolos 弹跳等特殊模式均在引擎 C++ 里）。社区/实测共识（AoE3 Heaven、ESOCommunity、Fandom）：

- 主目标：完整单次伤害（含弹丸合并），**不计入** DamageCap 池
- 溅射：以主目标为圆心、`damagearea` 为半径；**DamageCap 限制的是溅射能造成的总伤害**
- 多数单位 `damagecap ≈ 2×` 该动作的 `damage`，但**以 protoy 为准**（如迫击炮 `BarrageAttack`：damage 30、cap 90）
- 原版在范围内**按距离衰减**、目标选取也不完全是「均匀随机」；部分单位（如 Bolas）有独立弹跳逻辑

#### 斗蛐蛐简化 AOE 模型

一维场地 + 排布阵型（同排重叠），在 **DamageCap / DamageArea 读真实 protoy 值** 的前提下，仍简化选目标方式：

- 有 AOE 属性的兵种攻击时：
  - 主目标受 **100% 完整伤害**
  - **额外溅射目标数上限 = round(aoe_radius)**（四舍五入）
    - 例：鹰炮 aoe_radius=3 → 最多额外溅射 3 个目标 → 共 4 个受伤
  - 溅射目标从**与主目标距离 ≤ aoe_radius 的存活敌方个体**中随机选取
    - 同方同兵种个体距离=0，天然在范围内
    - 不同兵种需判断两个兵种位置间距离是否 ≤ aoe_radius（**后排离得远的兵种不会被溅射到**）
  - **DamageCap**：与本次出手**同一槽位**的 `damage_cap_*`（parser 读 `damagecap`）；JSON 无该字段时 fallback `合并基础攻 × 2`（有 proto 值时不得用 2× 覆盖）。**说明**：2× 非 protoy 写死的引擎公式，而是「有 cap 时中位数≈2× + 社区描述 + 旧模拟器」；缺 cap 的 49 条多为战役动作，见 `docs/aoe3-damagecap-audit.md` §2
  - 溅射伤害均匀分配：每个溅射目标 = `min(DamageCap / 实际溅射人数, 合并基础攻)`，再乘倍率/抗性
  - 溅射伤害同样应用倍率和抗性
  - 主目标的超杀伤害不转移
  - **未实现**：距离衰减、锥形/圆形精确几何、Bolos 弹跳等非 protoy 字段逻辑
- `aoe_radius` 字段缺失或为 0 的兵种 → **单体攻击**，不走 AOE 流程
- 兵种实际有 AOE 但字段缺失 → **数据 bug**，列入 §四数据缺失清单修复
- 模拟器**不做兜底猜测**（不能“看到是炮就猜 aoe=3”，数据是什么就是什么）

### 3.9 攻击分类（解析脚本核心逻辑）

**数据源**：游戏文件 `protoy.xml` 中每个 `<unit>` 的 `<protoaction>` 元素，
每个 protoaction 的 `damage / damagetype / maxrange / minrange / rof / damagearea / damagebonus*`
**整体绑定**——同一个 protoaction 的所有字段是一套真实游戏数据，铁律：**不可拆分拼接**。

#### 核心概念：攻击模式 vs 伤害类型（正交！）

这是两个**完全独立的维度**，**绝不能混淆**：

| 维度 | 取值 | 由谁决定 | 影响 |
|------|------|---------|------|
| **攻击模式（动作名）** | `HandAttack` / `RangedAttack` / `BuildingAttack` 等 | 引擎按动作名硬编码绑定行为 | 动画、弹道、是否停步、是否触发闪避 |
| **伤害类型** | `Ranged` / `Hand` / `Siege` | protoaction 的 `damagetype` 字段 | 吃哪种护甲（armor_ranged / armor_melee / armor_siege）|

**关键认知**：AoE3 引擎**不存在"远程/近战"的概念划分**。引擎只知道当前要执行哪个动作名
（Defend/Stagger/Volley 姿态 × Hand/Ranged 距离 × Building/Rocket/Guardian 等特殊），
每次激活一种攻击模式就行。我们斗蛐蛐模拟器的 ranged/melee/siege 三槽位划分是**人为简化**，
不是游戏本身的机制。

**举例**（伤害类型属于具体 protoaction，与该 protoaction 是远程还是近战无关）：

- ✅ **远程攻击 + Siege 伤害**：鹰炮主炮（`CannonAttack, damage_type=Siege`）
- ✅ **近战攻击 + Siege 伤害**：沙漠突袭者（`MeleeHandAttack, damage_type=Siege`）
- ✅ **远程攻击 + Hand 伤害**：战斧兵投斧、飞刀手（`RangedAttack, damage_type=Hand`）
- ✅ **近战攻击 + Hand 伤害**：绝大多数近战兵（默认情况）
- ✅ **全 Siege 多模式**：火兵有 10 种攻击模式全部 `damagetype=Siege`

> 因此 parser 的"槽位归类"和"伤害类型识别"是**两套独立判断**：
> - 归 melee/ranged 槽位 → 看**动作名标签**（引擎行为的唯一真实依据）
> - `damage_type_*` 字段 → 直接抄 protoaction 的 `damagetype`（决定它打哪种护甲）
>
> **绝对禁止**用 `damagetype` 来判断攻击归属哪个槽位。

**实现位置**：`scripts/crawler/aoe3_gamedata_parser.py :: _parse_attacks`



**设计原则**：
- **type / multipliers.vs 直接存游戏原始标签**（`AbstractCavalry`、`AbstractHeavyInfantry` 等），不翻译
- **倍率天然匹配**：`damagebonus.type` 和 `unit.type` 来自同一份 XML 同一套字符串，倍率匹配 = 字符串相等比对，不需要映射表
- **翻译只在展示层**：`i18n_zh.json` 负责 `AbstractXxx → 中文`，与数据/匹配逻辑解耦

#### 第一步：分类 ranged / melee / siege

每个 protoaction 按以下规则归入三个槽位之一（**只看动作名 + 射程兜底，不看 damagetype**）：

1. 动作名含 `BuildingAttack` → **siege**（拆建筑专用）
2. 动作名含 `HandAttack` → **melee**（引擎近战行为，不管 maxrange 和 damagetype）
3. 动作名含 `RangedAttack` → **ranged**（引擎远程行为，不管 damagetype）
4. 以上均不匹配时，按射程兜底：`maxrange < 6` → melee，`maxrange >= 6` → ranged

> 为什么信任动作名？因为 AoE3 引擎通过动作名硬编码绑定行为（动画/弹道/停步等），
> 这些名字是引擎代码里的字符串常量，不可能随意命名。射程只是参数，不驱动行为。
>
> 兜底阈值 6 的依据：AoE3 近战武器最大 maxrange 约 4~5（流星锤兵 4.0、翼骑兵 3.75），
> 最短远程射击从 6 开始（火绳枪骑兵 6.0）。

#### 第二步：选代表攻击

每个槽位可能有多个候选攻击（如鹰炮的 `CannonAttack` / `BombardAttack` / `CaseShotAttack`
都属于 ranged）。按 `ATTACK_PRIORITY` 常量选**优先级最高的一条**作为该槽位的代表，
该 protoaction 的所有字段（damage / range / rof / aoe / multipliers）整体写入对应字段族。

优先级设计原则（**Volley/Stagger > 具名主攻击 > Defend**；Defend 为手动防御阵型）：

- **远程**：`VolleyRanged` / `StaggerRanged`（统一齐射 > 交错；多数纯骑兵 protoy 仅有交错动作，自然落回）→ 具名主攻击（`BowAttack` 等，多数单位无齐射/交错后缀）→ `DefendRanged`（置底）
- **近战**：`VolleyHand` / `StaggerHand` → 具名（`MeleeHandAttack` / `BayonetAttack` / `HandAttack`）→ `DefendHand`（置底）
- **炮兵**：`BarrageAttack > RepeatingAttack > CannonAttack > …`（独立表）
- 具名表已核实（2026-05-29）：`SharpshooterAttack` / `CrackshotAttack` / `SwashbucklerAttack` 为英雄技，parser 排除；`BowAttack` / `RifleAttack` 等常态主武器保留。审计脚本 `scripts/aoe3_named_attack_audit.py`

**外部参考**（人工核对单位数值）：[AOE 3 Home City — 军事单位](https://aoe3homecity.com/zh-CN/units?type=military)（如[雇佣兵列表](https://aoe3homecity.com/zh-CN/units?type=military&tags=Mercenary)）

> **铁律**：只能整体选用一个 protoaction，不能把 A 的 damage 和 B 的 multipliers 拼在一起。

#### 写入 units.json 的字段

```
attack_ranged       = ranged 代表攻击的 damage
range               = ranged 代表攻击的 maxrange
range_min           = ranged 代表攻击的 minrange
rof_ranged          = ranged 代表攻击的 rof
damage_type_ranged  = ranged 代表攻击的 damagetype
aoe_radius_ranged   = ranged 代表攻击的 damagearea
damage_cap_ranged   = ranged 代表攻击的 damagecap（0 则模拟器 fallback 2×合并基础攻）
multipliers.ranged  = ranged 代表攻击的 damagebonus 列表

attack_melee        = melee 代表攻击的 damage
range_melee         = melee 代表攻击的 maxrange（近战射程，0 表示用默认 1.5）
rof_melee           = melee 代表攻击的 rof
damage_type_melee   = melee 代表攻击的 damagetype
aoe_radius_melee    = melee 代表攻击的 damagearea
damage_cap_melee    = melee 代表攻击的 damagecap
multipliers.melee   = melee 代表攻击的 damagebonus 列表

attack_siege        = siege 代表攻击的 damage（拆建筑专用）
range_siege         = siege 代表攻击的 maxrange
rof_siege           = siege 代表攻击的 rof
```

> **siege 槽位仅用于拆建筑展示，斗蛐蛐模拟器不使用**——一维场地上没有建筑可拆。
> 角色卡（`_atk_summary`）也不显示 `attack_siege`，避免误导群友（详见 §8.5）。

#### 战斗池准入规则（重要）

`Unit.has_attack` 只看 `attack_ranged` / `attack_melee`，**不算 `attack_siege`**：

```python
def has_attack(self) -> bool:
    return bool(self.attack_ranged or self.attack_melee)
```

**含义**：一个单位如果**只有 attack_siege 没有 melee/ranged**，会被自动排除出战斗池
（押注模式 `_build_*_pool` 和单挑模式都用 `has_attack` 过滤）。这是合理的——
它在斗蛐蛐里没有可拆的建筑，等于站桩。典型代表：
- 木制牛 `deeggwoodcattle`（彩蛋单位）
- 审判官 `desalooninquisitor`（治疗师，只有拆建筑攻击）

> ⚠️ **副作用警告**：如果一个真正能打兵的单位被 parser 误归类（如近战攻击
> 因为动作名不含 `HandAttack` 且 maxrange >= 6 被丢进 ranged 桶），它的 melee 槽位
> 可能为空。修法是**在 `ATTACK_PRIORITY` 中加入该动作名**或调整兜底阈值，
> **不要**在 `has_attack` 加例外白名单。



#### simulator 侧

- 战斗模拟只读 `attack_ranged` / `attack_melee` 及其对应字段族
- `damage_type_ranged` / `damage_type_melee` 决定吃哪种护甲：
  - `Siege` → `armor_siege`（极少见）
  - `Hand` → `armor_melee`
  - `Ranged` → `armor_ranged`
- 例：鹰炮 `attack_ranged=80, damage_type_ranged=Siege` 打火枪手（无 siege 抗性）→ 全额伤害

| 兵种 | 选中的 protoaction | 写入 units.json |
|------|-------------------|----------------|
| 鹰炮 falconet | `Cannon Attack`（damage=80, range=26, type=Siege, aoe=4） | attack_ranged=80, range=26, damage_type_ranged=Siege, aoe_radius_ranged=4 |
| 火枪手 musketeer | `Ranged Attack`（damage=23, range=12, type=Ranged） + `Hand Attack`（damage=13, type=Hand） | attack_ranged=23, attack_melee=13 |

### 3.10 单位改良（科技加成）

> **Status**: ✅ 已实现（2026-05-29）。数据源 `data/aoe3/raw/techtreey.xml` → 离线生成 `seeds/aoe3/unit_upgrades.json`；运行时由 `src/plugins/aoe3/upgrades.py` 的 `apply_upgrades()` 按时代叠加。

#### 3.10.1 档位是「累加」不是「替代」也不是「连乘」

精锐 / 禁卫 / 帝王是**前置链**（帝王 `prereq` 要求已研禁卫，禁卫要求已研精锐），所以升到高档时低档**同时生效**。它们的 effect 多为 `relativity="BasePercent"`——**每档按「基础值」算增量，增量相加**：

```
升到帝王火枪 = 基础 × (1 + 0.20 + 0.30 + 0.50) = 基础 × 2.00 = 200%
```

不是替代（×1.5），也不是连乘（×2.34）。火枪兵示例（基础 HP150 / 远程23 / 近战13）：

| 档位 | 系数 | HP | 远程攻 | 近战攻 |
|------|------|-----|--------|--------|
| 无 | ×1.00 | 150 | 23 | 13 |
| 精锐 Veteran | ×1.20 | 180 | 27.6 | 15.6 |
| 禁卫 Guard | ×1.50 | 225 | 34.5 | 19.5 |
| 帝王 Imperial | ×2.00 | 300 | 46 | 26 |

> 「指定某档」= 把该兵种**从无到该档的所有**逐时代升级（可研究 `UpgradeTech` + 通用 `Shadow` 自动档，见 3.10.5）增量按下表规则合进去，**不是只取一条**。

**各类别血/攻累加曲线**（实测自 `techtreey.xml`，含 `Shadow` 自动档）：

| 类别 | 2时代 | 3时代 | 4时代 | 5时代 | 说明 |
|------|-------|-------|-------|-------|------|
| 标准步/骑 | 100 | 120 | 150 | 200 | 精锐/近卫/帝王（可研究） |
| 散兵 | 100 | 120 | 150 | 200 | 精锐为 `Shadow` 自动档，余同标准 |
| 炮兵 | 100 | 100 | 125 | 175 | 无精锐/近卫；age4 `FieldGun`+25、age5 `Imperial*`+50 |
| 土著战士 | 100 | 120 | 150 | 200 | age3/4 绑文明逐兵链，age5 议会 `ImpLegendaryNatives`+50 |
| 亡命徒 | 100 | 120 | 150 | 200 | 酒馆精锐 + `Shadow` 近卫/帝王（见 3.10.4） |
| 佣兵 | 100 | 100 | 100 | 150 | 全程无逐时代档，仅升帝王政客 +50 |

#### 3.10.2 范围：按 relativity 决定数学，按 subtype 映射字段

扫描 525 个 `UpgradeTech` 的 effect 分布后，**做不做取决于 `relativity`**（它决定怎么算），能干净映射到现有 `Unit` 字段的都做；解锁 / 经济 / 美术不做。

| subtype | 映射字段 | relativity | 累加规则 | 做 |
|---------|----------|-----------|----------|----|
| `Damage` | `attack_*`（带 `action=`，按动作名归槽） | BasePercent | 增量相加 | ✅ |
| `Hitpoints`/`HitPoints` | `hp` | BasePercent | 增量相加 | ✅ |
| `MaximumRange` | `range`（带 `action=`） | Absolute | 固定值相加（如 +2） | ✅ |
| `DamageBonus` | `multipliers`（缩放该兵**已有**克制倍率，非加新条目） | 主要 Absolute | 每条**正倍率(≥1)** += amount；惩罚项(<1)不动 | ✅ |
| `DamageArea` | `aoe_radius_*`（带 `action=`） | Absolute | 固定值相加 | ✅ |
| `RateOfFire` | `rof_*` | **Assign** | **覆盖（不累加）** | ✅ 语义不同 |
| `MaximumVelocity` | `speed` | 混合(Absolute/BasePercent/Assign) | Abs 加 / BP 乘 / Assign 覆盖 | ✅ |
| `ArmorSpecific` | `armor_*` | Absolute | 固定值相加 | ✅ |
| `Cost`/`TrainPoints`/`BuildPoints`/`WorkRate`/`ResearchPoints` | 经济 | — | — | ❌ 无意义 |
| `UpdateVisual` | 换模型 | Absolute | — | ❌ 纯美术 |
| `Enable`/`ActionEnable`/`FreeHomeCityUnit`/`Lifespan` | 解锁 | — | — | ❌ 奇特效果不做 |

**relativity → 累加规则**（引擎核心，约一张表）：

- `BasePercent` → 各档增量**相加**（Damage / HP / Velocity）；
- `Absolute` → 固定值**相加**（Range +2、AOE、Armor、DamageBonus 正倍率）；
- `Assign` / `Override` → **覆盖**，多档时最后一档生效，**不累加**（ROF、Cost）；
- `Percent`（仅 3 例）→ 乘当前值（多档时连乘）。

> **`DamageBonus` 语义**：effect 形如 `subtype="DamageBonus" unittype="<vs>" amount="X"`——给「该兵对 `<vs>` 的倍率」加 X。规则=只对该兵**已存在的正倍率(>1)** += amount（不新建条目、不碰惩罚倍率<1）。`allactions=1` 落全槽，否则按 `action` 匹配代表动作落槽。

##### 落地方式：「整包随代表科技」而非全局扫 stat（关键修正 v15）

range/aoe/rof/速度/护甲/倍率 **不单独全局扫描**，而是**每条已被 §3.10.5 选中的代表科技（tier 线/Shadow），把它自带的这些 effect 一并整包提取进该档**——完全对齐铁律「整包进出」。理由：

- 这些 effect 多为 `Absolute`(delta) / `Assign`(覆盖) / `BasePercent`(倍率)，**不是**血攻那种 BasePercent 累乘；按 relativity 分别处理：`Absolute`→沿链累加 delta、`Assign`→取 ≤age 最新覆盖、`BasePercent`→累乘。
- **绝大多数 `DamageBonus`/`MaximumRange` 是兵工厂可研发科技（Rifling/HeatedShot/Paper Cartridge…）和土著舞蹈** —— 它们是玩家主动选择、无「N 时代自动生效」语义，属 §3.10.4 暂缓的「广谱兵工厂白名单」。**只纳入 tier 线/Shadow 自带的**，独立兵工厂研发科技仍暂缓。
- 落地结果（有界）：`range_add` 命中 40 档（阿布枪 +1/2/4、azap +2/4/6、皮革炮…）、`speed_add` 2 档（皮革炮 +0.5/1.0）、`mult_add` 10 档（投石/弓骑 对火炮 +0.5）；`aoe/rof/armor` 因只来自兵工厂/文明 age-up 科技 → **0 命中**（符合暂缓约定）。

##### 脏数据护栏（v15 复核结论）

DE 部分科技含**离谱占位值**，无法靠语义识别（6074 条），用「合理上限」丢弃：

- 例：`DEEliteSlingersShadow` 给投石手 `VolleyRangedAttack` **+147 射程**（同 tech 里其余动作才 +7）——代表动作恰是 Volley 会漏进来，被 `SANE_CAP` 丢弃。
- 上限：射程 ≤10、AOE ≤4、速度 ≤12、护甲 ≤0.5、倍率 ≤5（正常改良远小于此）。
- **削弱档复核**：血攻 `<1` 的增量是兵工厂护甲副作用/置换（如 azap 胸甲 −10%），**不是 tier 线**；tier 线（`DEVeteran/Guard/ImperialAzaps` +20/30/50）另被正确捕获 → azap = 100/120/150/200。丢弃负增量正确。
- **action 去重**：同一 effect 常对 Volley/Defend/Stagger/BuildingAttack 各出一条；**只取等于 `protoaction_*` 的代表动作那一条**，绝不四条相加（铁律）。
- 全量扫描：对所有有攻击单位 ×age{3,4,5} apply 后，**升级造成的越界 = 0**（基础数据自带的海军 x10/间谍 x40 等不在升级范围内，未被触碰）。

#### 3.10.3 与攻击数据铁律的关系（天生支持的原因）

`Damage` / `MaximumRange` / `DamageArea` / `DamageBonus` 这些 effect **都带 `action="..."`**，按**动作名**作用——正好对齐 `.cursor/rules/aoe3-attack-data-design.mdc`：`units.json` 已按代表动作整包存了 `protoaction_ranged/melee`，所以「给某攻击类型加射程 / AOE / 倍率」只改对应槽位，不破坏 ranged/melee 正交。

**约束**：tech 的 `action` 必须等于 parser 为该槽选定的**代表动作**，否则该条改良**不命中**（静默跳过，符合铁律——不另选动作拼接）。

#### 3.10.4 匹配方式：按 id 精确匹配（兵种升级）vs 按标签（全局，不做）

- **兵种升级**：effect 的 `<target type="ProtoUnit">Musketeer</target>` 与 `units.json` 的 **`id`**（= `internal_name.lower()`）精确匹配（`Musketeer`→`musketeer`）。验证：450 条 unit-only 改良覆盖 **268 个**有攻击的战斗单位；未匹配目标仅 16 个，全是非战斗对象（`Hero`/建筑/附着物），安全忽略。**这是精确名匹配，不是标签匹配。**
- **三大类「类别科技」（标签匹配，做）**：土著 / 亡命徒 / 佣兵在游戏里有**按大类**(非按兵)的战斗强化，标签匹配很干净（我们单位分别带 `AbstractNativeWarrior` / `AbstractOutlaw` / `Mercenary`，数量 native 系 / 53 outlaw / 66 merc），按时代给 +X% 血攻（`BasePercent` 累加）：

  | 亚洲寺院 | `ypMonasteryCompunction` | 3 | +20% | 可研究 |
  | 非洲 | `DEImprovedOutlaws` | 3 | +20% | 可研究 |
  | （接 WildWest） | `DESaloonPeacemakers` | 4 | +10% | 可研究 |

- **其余广谱兵工厂科技 → 已实现为「通用科技 roguelike」（§3.10.6b）**：从 68 条 `Abstract*` + 卡片 + RG 里手工精选 66 条精华池（按 scope+effect 去重），以 roguelike 随机方式叠在 tier 之上。详见 §3.10.6b。

#### 3.10.5 选链规则：沿一条 prereq 链累加，优先通用线

同一兵种在同一时代常有**多条互斥升级变体**（通用线 vs 文明皇家卫队 RG vs 革命 Rev）。例：火枪兵 age4 有 `GuardMusketeers`（通用）+ `RGRedcoats`（英）+ `RGGuerreiros`（葡）。**禁止 age≤N 全加**（会把互斥变体一起叠、数值爆炸）。规则：

1. **逐时代独立选一条**：对每个时代 age∈{3,4,5}，在该兵种该时代的候选升级里**只选一条**，把其增量按 3.10.2 规则累加进选定时代之前的所有档。
2. **候选 = 可研究 `UpgradeTech` ∪ 通用 `Shadow` 自动档**（⚠️ 关键，曾漏）：部分兵种的精锐/近卫/帝王**不是可研究科技，而是 `Shadow` 隐藏档**（升到对应时代由引擎自动激活、UI 不显示），它们就是该兵的逐时代档，**必须纳入**。例：
   - 散兵精锐 = `VeteranSkirmishersShadow`（`prereq=Fortressize`，age3 +20%）→ 散兵实为标准 **100/120/150/200**（先前误算 180%，因只扫了 `UpgradeTech`）；
   - 亡命徒近卫/帝王 = `DEGuardOutlawShadow`(age4 +30)、`DEImperialOutlawShadow`(age5 +50)。
3. **必须按 prereq 解出该 `Shadow`/`UpgradeTech` 的时代并过滤文明专属**：只保留 prereq 为通用 `Fortressize`/`Industrialize`/`Imperialize` 的；**排除带文明前缀/无 age 前置的 `Shadow`**（`Age0<文明>`、`Imperialize<文明>`、`*AfricanShadow`、`Swedish*Shadow`、`DEAge0*`、`YPAA*StartingTechs` 等都是文明开局/专属加成，不进通用线）。同一兵同档若同时有可研究与通用 `Shadow`，二选一**不重复计**。
4. **优先通用线** `Veteran→Guard→Imperial`（含 `DEVeteran*`/`DEGuard*`/`DEImperial*` 及对应 `*Shadow` 等仅前缀/兵名/可见性不同的同构线）。
5. **回退文明变体**：若**某一时代没有通用线**，该档取文明变体 RG（例：长矛骑兵 `lancer` age4 用通用 `GuardLancers`、age5 无通用 → 回退 `ImperialGarrochistas`）。**逐档独立**，允许某兵 age4 通用 + age5 RG 混搭（无文明系统下的近似）。
6. **排除革命**（关键：革命本不该出现，是「搭便车」混入的）：tier 自动档靠 `Shadow` 标记识别（散兵精锐 = `VeteranSkirmishersShadow`，无研究按钮、必须认 `Shadow` 才捞得到），而**革命隐藏档也复用了 `Shadow` 标记**（`Rev*`/`DEREV*`/`DEHCREV*`），于是混进候选池——按前缀排除。同理开局 `Age0*`/`*StartingTechs`、文明专属 `*AfricanShadow` 等也是复用 `Shadow` 混入的噪音，一并排除。**「提纯 tier 线」= 认 `Shadow`/`UpgradeTech` 捞档 → prereq 解时代 → 排除革命/开局/文明专属/政客**，这是本功能脏数据处理的主体（`Shadow` 标记不纯所致）。
7. **2 时代 = 无军事改良**：普通战斗兵种的通用军改（含 `Shadow` 自动精锐）最早在 age3（`Fortressize`）解锁；age2 仅 1 条战役民兵改良，可忽略。**禁止 age≤N 把同档多条变体一起加**（否则通用+RG 叠加、数值爆炸）。

> ⚠️ **无文明系统的代价（记录给未来）**：英国红衣兵 `RGRedcoats` 等皇家卫队在游戏里强于通用禁卫，本方案按通用值算（略低）。**待将来加文明系统时**，可按对局文明选择对应 RG/革命变体链，替换通用线——届时选链规则的第 2/3 步改为「按 civ 选支」。

#### 3.10.6 指令、时代与展示

- **指令**：`斗蛐蛐 N时代`（N ∈ **2~5**）指定本局时代；可与现有模式/预算组合（如 `斗蛐蛐 自选 火枪 散兵 5时代`）。**所有模式**都接受该参数（§④）。
- **对称**：双方**同档**（同一时代），两边一起缩放，LCM/王中王平衡基本不被破坏。
- **兵池按时代限定**：本局只抽 `unit.age ≤ N` 的兵（751 个战斗兵全有 `age`，0 缺失；Exploration1/Commerce2/Fortress3/Industrial4/Imperial5）。早登场单位（如瑞典皮革炮 age2、长弓 age2、廓尔喀 age2、age2 龙骑/弓骑）本就是**文明早登场特权**，照常出现。**不**因低时代兵变少而做特殊过滤规则。
- **默认**：不写时代参数 = **3 时代**（693 兵，含散兵/鹰炮等经典兵；通用军改从 age3 起，故默认就有改良）。
- **累计可用兵数**：age1=131 / ≤age2=516 / ≤age3=693 / ≤age4=755 / ≤age5=756。
- **预先结算**：开战前就把改良算进 `Unit` 副本（HP/攻/射程/AOE/倍率…），模拟器与兵种信息卡都用**加成后**数据；`_create_soldier` 不感知科技。
- **兵种信息卡**：直接显示**加成后**面板，**不逐条列**兵种独立升级（精锐/禁卫/帝王），避免冗余。
- **押注简报**：在简要对局信息里写明**本局时代**（如「⏳ 本局：帝国时代（5）」）。
- **类别科技展示**：在押注简报的时代之后，按本局是否含土著/亡命徒/佣兵单位，逐条列出已激活的类别加成（文本以 stringtable 实际译名为准）：

```
本局：帝国时代（5）
已激活类别科技：
  · 传奇土著战士：土著 血/攻 +50%
  · 雇佣兵契约：佣兵 血/攻 +50%
```

#### 3.10.6b 通用科技（roguelike 横向加成，已实现）

> **Status**: 已实现。群配置开关（默认关），指令 `斗蛐蛐 科技开` / `斗蛐蛐 科技关`。

**定义**：通用科技 = 兵工厂/教堂研发 + 本城卡片 + 文明近卫 RG 等**横向战斗增益**，区别于上述 §3.10.1~§3.10.5 的纵向 tier 成长（精锐→近卫→帝王）。

**数据来源**：`seeds/aoe3/generic_techs.json`（由 `scripts/crawler/aoe3_generic_techs_parser.py` 离线生成），66 条去重后的精华池，全部能匹配到 `units.json` 真实单位。

**池子构成**：
- **兵工厂广谱**（age2）：步兵胸甲、骑兵胸甲、燧发枪、纸包弹、职业炮手、炮架耳轴、军乐鼓手、半回旋、来复线、燃烧弹
- **教堂特色/双刃**（age2~3）：细红线★（火枪 +20%血 −10%速）、钢甲★（重步 +25%血 −15%速）、蒂利的纪律（步兵 +20%速）、第二保证、卡皮库鲁
- **本城战斗力卡**（age3，按 scope+effect 去重取最强）：骑兵/步兵/炮兵/远程骑兵战斗力 +15%血攻
- **特色具体兵**：旧朝改革★（中国连弩/长矛/怯薛/草原骑兵 +50%血攻）
- **RG 近卫**（age4）：红衣军→火枪、铁骑→枪兵、撞针枪→散兵 等 +10%血攻（叠在 tier 近卫之上）

**排除**：
- 印度/亚洲单位"纪律严明/光荣/高贵"（是纵向 tier，非横向通用）
- 原住民盟友 `Nat*` 战士团（冷门地图土著）
- vs建筑/船伤害（场上无对应目标）、奇观/英雄/建筑
- 土著/亡命徒/佣兵类别（已由 §3.10.4 类别科技模块处理）
- Team 队伍卡（数值低且与战斗力卡重复）

**每局随机规则**：
- **每方各 K/2 条**：单挑/自选 K=2 → 每方 1 条；押注 K=4 → 每方 2 条
- **相关性**：科技 scope ∩ 己方单位 type 非空才入候选池
- **age 门槛**：`tech.age ≤ 本局 age`
- **双方独立抽取**：可能抽到同一条（共用 → 白赚）
- **叠加顺序**：先 tier（`apply_upgrades`），再通用科技（`apply_generic_techs`）

**开关**：
- `斗蛐蛐 科技开` → 群配置存 `aoe3_battle.generic_techs=on`
- `斗蛐蛐 科技关` → 群配置存 `off`
- **默认关**，黑名单乱斗不启用

**战报展示**：押注简报在类别科技之后显示本局抽到的通用科技，格式：
```
🔬 本局通用科技（roguelike）：
   🔴 骑兵胸甲（HeavyCavalry: 血+10%）
   🔵 细细的红线（musketeer: 血+20%, 速-10%）
```

#### 3.10.7 实现分层（落地约定）

- **离线**（parser，无需游戏）：扫 `techtreey.xml` 生成 `seeds/aoe3/unit_upgrades.json`，结构按 id 索引、按时代给出选定链的合并增量，例如：

```json
{
  "musketeer": {
    "3": { "hp_mult": 1.20, "damage_mult": 1.20 },
    "4": { "hp_mult": 1.50, "damage_mult": 1.50 },
    "5": { "hp_mult": 2.00, "damage_mult": 2.00 }
  }
}
```

> **已落地字段**（v15，cumulative 预合并；只挂在逐兵 `units` 上，类别表仅血/攻）：
> `hp_mult` / `damage_mult` / `range_add{ranged,melee}` / `aoe_add{...}` / `rof_set{...}`（覆盖）/ `armor_add{melee,ranged}` / `speed_add` / `speed_mult` / `speed_set` / `mult_add{slot:{vs:delta}}`。选链与 relativity 累加在**生成期**完成，运行时只做乘/加/覆盖。

- **类别科技**（土著/亡命徒/佣兵）另存一张小表（按标签 × 时代给 +X%），运行时对带对应标签的兵叠加。
- **逐兵 vs 类别 = 按时代取较大者整包（`max`，v15 修正）**：不再「逐兵优先否则类别」。原规则会让**共享单位的小额文明逐兵档顶掉更大的类别加成**（瑞士长枪有荷兰 Waardgelders +10% 逐兵档，5 时代本应吃佣兵 +50%，旧逻辑只给 +10%）。改为逐兵链与类别各取该档条目、取 hp_mult 较大的整包：既修瑞士长枪、又稳解土著「逐兵传奇 +50 与类别传奇 +50」的 double（取 max=一份，不叠成 +100）。
- **运行时**：`apply_upgrades(unit, age) -> Unit`（`dataclasses.replace` 出副本，不改全局 seed；多倍率列表新建，不原地改），在进 `BattleSimulator` 前、阵容生成后应用；血/攻取 max 整包，range/aoe/rof/速度/护甲/倍率取逐兵链整包；兵种卡用副本渲染。
- **兵池过滤**：阵容生成入口按 `unit.age ≤ N` 过滤候选池（各 `get_*_pool` 加时代上限参数）。

## 六、独立测试模块（开发期工具）

为了便于在不卷入押注/经济/群消息的情况下专注调试战斗模拟器本体，需要一个**独立可运行的纯模拟测试入口**：

### 6.1 定位

- **不是**最终游戏的一部分，是开发期的**模拟器单元演练工具**
- 阉割掉押注、经济、群消息、状态机等一切非"模拟"内容
- 直接：`选两个兵种和数量 → 跑模拟 → 打印事件流和结果`

### 6.2 入口

- 脚本位置：`scripts/aoe3_battle_sim.py`（独立 CLI，与 `play_cli.py` 解耦）
- 调用方式：`uv run python scripts/aoe3_battle_sim.py`
- 提供三种运行方式：
  - **交互式**：菜单选兵种 → 输数量 → 跑
  - **参数式**：`--red musketeer:10 --blue pikeman:8`，便于回归测试和脚本批跑
  - **靶机模式**：`--dummy musketeer:5`，蓝方为不动不攻击的高 HP 靶机，用于验证 windup/DPS 等数据
    - `uv run python scripts/aoe3_battle_sim.py --dummy musketeer:5 --debug`
    - `uv run python scripts/aoe3_battle_sim.py --dummy longbowman:10 --dummy-hp 50000 --dummy-count 3`
    - 输出：首发时间、预期首发（移动+windup）、总伤害、DPS、各士兵首发 tick

### 6.3 输出

- 完整事件流（带时间戳）逐行打印
- 最终战报：胜方、剩余兵力、战斗时长、各方击杀数、伤害统计
- **不接入 QQ 群消息、不接入金币系统、不走状态机**

### 6.4 复用关系

- 测试模块直接 import 真实的 `simulator.py`（`src/plugins/games/aoe3_battle/` 下的核心模拟器）
- 即测试模块和正式游戏共用同一份模拟器代码，确保"测得过 = 跑得通"

## 七、调试日志规范（开发期强制要求）

> **背景**：电子斗蛐蛐是项目里规模最大、状态最复杂、最容易出 bug 的功能（一维场地 + 个体 HP + 远近切换 + 抗性倍率 + AOE + 事件流分批播报，任一环出错都会"看起来跑了但结果不对"）。  
> 因此 **要求开发期所有核心环节都必须打详细调试日志**，方便后续通过日志快速定位问题。

### 7.1 总体原则

- **宁多勿少**：开发期日志按"事无巨细"标准打，不要怕日志多
- **可开关**：通过日志级别（DEBUG / INFO / WARNING / ERROR）和命名 logger 控制输出
  - 默认 `DEBUG`：开发与独立测试器（§6）启用
  - 上线后调整为 `INFO`：群消息只看关键节点和异常
- **结构化优先**：日志内容包含关键字段（时间戳、单位 ID、坐标、HP、伤害值），便于 grep/正则筛选定位
- **可追溯**：每条日志带 tick 编号或模拟时间戳，能根据日志时间线还原战斗

### 7.2 必须打日志的关键环节

按战斗流水的发生顺序：

#### 阵容生成阶段
- `INFO` 兵种池筛选结果（数量、排除原因）
- `DEBUG` 每次随机抽兵种的候选与最终选择
- `INFO` 双方最终阵容（兵种、数量、总资源、人口、双方资源差异）
- `WARNING` 资源平衡多次重抽未达标时

#### 模拟器初始化
- `INFO` 战斗参数（场地长度、tick 间隔、最大时长）
- `DEBUG` 每个士兵的初始状态（id, 阵营, 兵种, HP, pos, 攻击力, ROF, 抗性）

#### 每 tick 主循环
- `DEBUG` tick 编号 + 模拟时间戳（每 tick 都打）
- `DEBUG` 移动事件：哪个兵种从 X 移到 Y，最近敌方距离
- `DEBUG` 攻击模式判定：`unit#A 距 unit#B = 4.2，选择=远程攻击`
- `DEBUG` 攻击执行：基础攻击、倍率链、抗性、最终伤害、是否触发贴脸 50%
- `DEBUG` AOE 触发：主目标 + 溅射目标列表 + 各自伤害
- `INFO` 死亡事件：`unit#A (火枪手) 阵亡，剩余 X/Y`
- `DEBUG` 目标重新锁定：`unit#A 目标 unit#B 阵亡，重新锁定 unit#C`

#### 异常与边界
- `WARNING` ROF 缺失走默认值
- `WARNING` 伤害类型字段缺失走默认推断
- `WARNING` 模拟到达最大时长，按剩余 HP 比判胜
- `ERROR` 出现负 HP / 死亡单位仍被攻击 / 锁定不存在的目标 等不应出现的状态

#### 播报层
- `INFO` 每个 2 秒窗口的事件摘要（即使被跳过也要 DEBUG 记录"窗口跳过原因"）
- `INFO` 关键节点触发（开战 / 首杀 / 半灭线 / 终结）
- `DEBUG` 话术模板渲染前后的数据对比

### 7.3 实现约定

- 使用 Python 标准 `logging` 模块，**不要** `print`
- Logger 命名遵循模块层级：
  - `aoe3_battle.lineup`（阵容生成）
  - `aoe3_battle.simulator`（模拟器主循环）
  - `aoe3_battle.simulator.combat`（攻击/伤害判定）
  - `aoe3_battle.simulator.aoe`（AOE 计算）
  - `aoe3_battle.broadcaster`（播报层）
  - `aoe3_battle.economy`（押注与结算）
- 独立测试器（§6）默认输出 `DEBUG` 到 stderr，便于直接看
- 战斗模拟结束时，把完整事件流 dump 到一个临时文件（如 `/tmp/aoe3_battle_<期数>.log`），即使群里没看到日志也能事后翻

### 7.4 日志卫生

- 不要在日志里打巨型对象（如完整士兵列表的 repr），用关键字段
- 不要在循环最内层打超过 1 行的多行日志
- 上线时通过配置文件统一抬高级别，**不要**为上线临时删日志（删了下次还要补）

## 八、播报话术规范

### 8.1 总基调

- **朴素纪实 + 偶尔皮一下**——不堆形容词、不史诗渲染、不过度解说
- 让群友脑补，文字只提供"事实锚点"：兵种、数量、距离、伤亡数
- "皮一下"指偶尔吐槽真实数据带来的反常现象（鹰炮一炮 8 杀、雇佣兵打疯了），靠节制的吐槽自带笑点，不硬塞段子

### 8.2 播报片段格式（每 2 秒窗口）

#### 标准片段（窗口内有伤亡）

```
⏱ 04-06s
🔵 火枪手齐射，🔴 散兵 -2（剩 8/10）
🔴 散兵反击，🔵 火枪手 -1（剩 9/10）
📊 战况 🔴 8/10 vs 🔵 9/10
```

#### 极简片段（窗口零星击杀）

```
⏱ 14-16s
🔥 拉锯：🔴 -1 / 🔵 -1
📊 🔴 3/10 vs 🔵 4/10
```

#### 格式约定

- "📊 战况"行**每个非沉默片段都带**，给群友提供"打到哪了"的锚点
- 兵种动作词按"该窗口该方造成的死亡数"动态选词（§8.5）
- 每条播报后 sleep 2 秒，节奏接近真实战斗

### 8.3 关键节点（独立成条优先播报）

| 节点 | 触发条件 | 文案模板 |
|---|---|---|
| **开战** | 战斗开始 | 单兵种：`⚔️ 战斗打响！🔴 [10 火枪手] vs 🔵 [8 长枪兵]` 多兵种：`⚔️ 战斗打响！🔴 [2黑骑士 + 3祖瓦夫 + 11火枪手] vs 🔵 [1铁甲舰 + 7火枪手 + 16散兵]` |
| **首杀** | 整场第一次有人阵亡 | `🩸 第 X.Xs，🔵 长枪兵率先溅血，🔴 火枪手 -1` |
| **重大伤亡** | 单次攻击造成 ≥ 阈值人死亡（阈值=max(4, 总人数×5%)），每窗口只保留击杀数最多的一条 | `💥 第 X.Xs，🔵 鹰炮一炮带走 🔴 火枪手 ×5！` |
| **半灭线** | 任一方剩余兵力首次 ≤ 50% | `⚠️ 🔴 已损失过半（剩 4/10），战况告急` |
| **终结** | 一方全灭 | `☠️ 第 X.Xs，🔵 长枪兵踏平最后一名敌兵` |

- "重大伤亡"比"首杀"更有冲击力，**两者都保留**（首杀负责开局气氛，重大伤亡负责高潮）
- 重大伤亡阈值 = `max(4, 总人数 × 5%)`，大规模对局自动提高门槛
- **每个 2 秒窗口内最多只发一条重大伤亡播报**（取该窗口内击杀数最多的一条），避免刷屏
- 重大伤亡和半灭线**可以同片段触发**（先发重大伤亡条，再发半灭线条）

### 8.4 沉默期填充（W4 决议）

> **背景**：连续多个 2 秒窗口零伤亡时，完全沉默会让群友以为机器人卡死。  
> 单兵种接近期短（4~5 秒），但**血厚单位**（如黑骑士 600HP、要塞 HP 几千）出现时，零伤亡可能持续 10 秒以上，必须有动静。

#### 触发规则

- 连续**第 2 个**零伤亡窗口起，发一条**填充消息**（注意：第 1 个零伤亡窗口仍保持沉默，避免吵）
- 之后每 2 个零伤亡窗口发一次（即每 4 秒一条），直到出现伤亡为止
- 填充消息**不 sleep**（与零伤亡窗口共用，不打断节奏）

#### 填充消息选词（按战况随机一句）

**接近期**（双方都在移动靠近）：
- `⏳ 双方仍在接近，剑拔弩张...`
- `⏳ 战鼓未响，杀机已生`
- `⏳ 红蓝双方逐渐靠近射程`

**对耗期**（双方都在攻击但都没死，典型血厚单位互殴）：
- `⏳ 双方激烈交火，暂无伤亡`
- `⏳ 装甲互啃，子弹打在肉身上像挠痒...`（皮一下）
- `⏳ 互相伤害中，HP 缓慢下降`
- `⏳ 重甲单位的较量，比的就是耐心`

**僵持期**（HP 都低于 30%）：
- `⏳ 双方都已残血，下一击可能就是终结`
- `⏳ 鏖战至此，胜负只在一念之间`

#### 实现要点

- 模拟器在事件流里给每个 tick 标注当前**战况阶段**（接近 / 对耗 / 僵持），由播报层据此选词
- 阶段判定（基于事件流统计，**不是**实时 HP）：
  - 接近期：尚无任何攻击事件
  - 僵持期：双方剩余 HP 比例都 < 30%
  - 对耗期：以上之外
- 同一阶段内的话术池**不重复抽**（避免连发两条一样的），抽完池子重置

### 8.5 动态选词（W7：按伤害分类）

击杀播报术语**根据实际攻击模式 + 伤害类型 + 单位标签**选词，不再按单位类型标签分类。

#### 分类维度

| 分类 key | 条件 | 术语风格 | 示例单位 |
|---|---|---|---|
| `ranged_gunpowder` | 远程 + Ranged伤害 + Gunpowder标签 | 枪炮 | 火枪手、龙骑兵 |
| `ranged_archer` | 远程 + Ranged伤害 + Archer标签 | 弓箭 | 长弓手、弓骑兵 |
| `ranged_normal` | 远程 + Ranged伤害 + 兜底 | 通用射击 | 休伦木盾等 |
| `ranged_siege_art` | 远程 + Siege伤害 + Artillery标签 | 大炮 | 鹰炮、臼炮 |
| `ranged_siege` | 远程 + Siege伤害 + 兜底 | 爆破/投掷 | 掷弹兵、骑马掷弹兵 |
| `melee_hand` | 近战 + Hand伤害 + 兜底 | 冷兵器 | 长枪兵、黑骑士 |
| `melee_siege` | 近战 + Siege伤害 | 火攻 | 沙漠突袭者 |

#### 判定流程

1. 从 DEATH 事件取 `killer_attack_mode`（ranged/melee/ranged_penalized）和 `killer_damage_type`（Ranged/Siege/Hand）
2. 先按攻击模式分：ranged 系 vs melee 系
3. 再按伤害类型分：Ranged / Siege / Hand
4. 最后用单位标签辅助细分（Gunpowder / Archer / Artillery）
5. 每层都有兜底，不会漏

#### 死亡数档位（每类共用）

| 死亡数 | 档位 key |
|---|---|
| 1 | `"1"` |
| 2~3 | `"2-3"` |
| 4+ | `"4+"` |

- 窗口内同一方多次攻击合并成一条播报（不要每次攻击单发，太碎）
- 词库写在 `phrases.py` 常量文件里，方便调整不动逻辑

#### 阵型 emoji（独立于术语系统）

阵型面板中的 emoji 仍按单位类型标签判定（不变），但优先级调整为：
**Ship > Elephant > Camel > Cavalry > Artillery/Siege trooper > 步兵细分 > 兜底**

骑兵标签优先于 `Siege trooper`，避免草原骑兵等被误显示为 💣。

#### 战前角色卡

`_atk_summary` 不再显示攻城攻击（`attack_siege`），因为斗蛐蛐模拟器不使用攻城攻击，展示会误导群友。

### 8.6 最终战报 + 结算（一起发，W3）

**单兵种战报示例：**

```
🏆 ━━━ 战斗结果 ━━━
胜方：🔵 蓝方（2号）
战斗时长：47.3 秒

🔴 红方 全军覆没
  · 火枪手 ×10 → 0

🔵 蓝方 残兵
  · 长枪兵 ×6（损失 2/8）

🎖 MVP：🔵 长枪兵 #3
   伤害 287 / 击杀 4 / 综合分 343.5

━━━ 押注结算 ━━━
押对：@张三 @李四 @王五（3 人）
每人 +5 金币
押错：6 人，每人 -5 金币
全员参与奖：+1 金币
```

**多兵种战报示例：**

```
🏆 ━━━ 战斗结果 ━━━
胜方：🔵 蓝方（2号）
战斗时长：38.7 秒

🔴 红方 全军覆没
  · 黑骑士 ×2 → 0
  · 火枪手 ×11 → 0

🔵 蓝方 残兵
  · 铁甲舰 ×1（存活，HP 1847/3000）
  · 火枪手 ×4（损失 3/7）
  · 散兵 ×12（损失 4/16）

🎖 MVP：🔵 铁甲舰 #1
   伤害 1520 / 击杀 6 / 综合分 1060

━━━ 押注结算 ━━━
押对：@张三 @李四（2 人）
每人 +5 金币
押错：3 人，每人 -5 金币
全员参与奖：+1 金币
```

#### MVP 评选

- **综合分 = 伤害 × 0.5 + 击杀 × 50**
  - 让"补刀王"（高击杀低伤害）和"输出王"（高伤害低击杀）都有机会
  - 系数可在常量配置里调
- 平局时按伤害排序作为 tiebreaker
- MVP 限胜方（败方再 MVP 也无意义）

#### 押注结算
- 押对玩家**点名**（@形式）；超过 5 人时显示前 5 + "等 N 人"
- 平局退还入场券，不评 MVP（双方都没赢）

### 2026-05-15 追加决议（阵型间距优化）

- ✅ **排间距调大**：`ROW_SPACING` 从 1.5 → 2.5，确保 range_min=2 的远程兵在前排倒下后仍处于有效射程内（距离 2.5 > range_min 2），不会立即被迫近战
- ✅ **去掉 FORMATION_MAX_DEPTH 限制**：原 max_depth=12 会压缩大队伍排间距（如 8 排压缩为 1.71），与加大间距的目标矛盾；去掉后排间距固定 2.5，自然展开
- ✅ **移动边界改为动态计算**：不再使用固定 max_depth 作为场地边界，而是根据初始化后的实际兵力部署位置计算
- ✅ **移除人口上限**：不再有 30 人口硬约束，个体数完全由预算决定

### 2026-05-15 追加决议（近战 CAP + 渗透机制）

- ✅ **近战攻击 CAP**：同一目标最多被 `MELEE_ATTACK_CAP=4` 个纯近战兵同时近战攻击，防止大量近战兵堆叠秒杀前排
- ✅ **渗透状态**：当纯近战兵（has_melee=True, has_ranged=False）发现所有贴脸目标 CAP 已满时，进入渗透状态：
  - 速度降为 `speed × 0.1`（INFILTRATE_SPEED_MULT），缓慢穿越前排
  - 不攻击（target_id=None），直到找到 CAP 未满的后排目标

- ✅ **战力分公式**（实现在 `lineup.py :: power_score`）：
  `HP_eff × DPS_eff`，其中 `HP_eff = HP / (1 - max_armor)`、
  `DPS_eff = max(dps_ranged × aoe_mult, dps_melee × aoe_mult)`、
  `aoe_mult = 1 + aoe_radius × 0.3`
  - **不计 `attack_siege`**：是 BuildingAttack 槽，模拟器不读（§3.9）
  - **不计倍率（multipliers）**：克制天然在战斗里发挥作用，避免加特林骆驼之类 ×100 倍率爆表
  - **不计 speed / 射程**：1v1 模拟里影响小且难量化
- ✅ **目标战力分锚定每方独立 `min × r`**（既不是 `max × r`，也不是全局 `min × r`！）：
  `red_target = min(red_scores) × r`，`blue_target = min(blue_scores) × r`，
  其中 `r ~ U(3.0, 6.0)`
  - **为什么用 min**：池内战力分跨度约 **360 万倍**（垮盛顿 80 亿 vs 民兵 2.8k），
    用 `max × r` 会让弱方堆几万个个体，模拟器跑爆
  - **为什么用本方独立 min**：用全局 min 时，若任一方抽到弱酋长（双月/民兵），
    target 锚到 11k，让另一方的英雄（曼苏尔/大名）被压成"1 只保底"——
    出现"3 双月 vs 1 曼苏尔"这种数量悬殊不好看。本方独立后变 5v5
  - `r` 即"本方最弱兵出兵数"。范围 3~6：避开 r=8 时节奏拖太长
- ✅ **抽兵权重 `[40, 40, 20]`**（1/2/3 兵种）：保留单兵种神仙打架的频率，比普通押注的 `[0, 50, 50]` 多了单兵种分支

- ✅ **战力分明牌**：面板展示每方总战力 + 每兵种单兵分，作为押注核心信息。群友自行判断"押稳赢"或"赌冷门"
- ✅ **不搞群友点将**：复杂度高，首期不做。等基础玩法稳定后再考虑
- ✅ **不搞同组对决 / 单挑模式**：玩腻了，乱斗就是要全池随机
- ✅ **不搞胜率/历史排行**：与 §8.7 "不实现"哲学一致
- ✅ **3 辆怪兽卡车暂时进不了池**：parser 把它们的攻击误归到 `attack_siege` 槽（`has_attack=False`），
