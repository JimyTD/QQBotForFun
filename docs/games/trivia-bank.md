# 趣味问答 · 题库方案设计（v1）

> 本文档记录"从纯 LLM 出题切换到题库驱动"的完整决策依据、实施方案和验收标准。
> **和 `docs/games/trivia.md` 的关系**：trivia.md 是游戏整体设计，本文档是**题库子系统**的 RFC 式专项设计。

---

## 0. TL;DR

- **从 v1.4 起，趣味问答改为纯题库模式**，运行时**零 LLM 调用**
- 离线用 LLM 批量生成**每类若干道题 × 2 套线索**，代码自检 + 人工抽查后固化到 `seeds/trivia_bank/<type>.json`
- 每类规模：country 80 / city 120 / food 120 / person 100 / animal 80 / idiom 80 = **580 道**
- 每答案预存 **2 套线索**，运行时随机抽一套 → 同答案再出现时"题面不同"
- **不保留 LLM 兜底**：题库跑光则本局按剩余题数跑完（或允许跨局重复抽取）

---

## 1. 决策背景：为什么抛弃纯 LLM 出题？

### 1.1 实测暴露的三类问题（2026-04-30）

| 问题 | 表现 | 日志证据 |
|---|---|---|
| **选词池窄** | 10 题里 5 题都是孙悟空；跨局也高度重复 | 用户 20:30 实测截图 |
| **self-leak 连环翻车** | person 类连续 3 次 retry 全栽在"答案/别名在自己线索里" | 20:30:27–20:31:30 日志 |
| **生成慢** | 单题 7-15s，整局 10 题 ≥ 2 分钟纯等 | `[llm] latency=17313ms` |

### 1.2 根因（不可靠 prompt 修复的结构性问题）

1. **训练数据头部偏置**：`glm-4-flash` 等小模型对"著名人物/国家"的词频分布极端偏头，温度再高也压不住。v1.3 的滚动 avoid 能逼它换，但**换出来的依然是头部里剩下的几个**，玩 3 局还是会撞
2. **名字 = 作品名陷阱**：哈利·波特、堂吉诃德、福尔摩斯、宙斯…… 这类答案**天然容易自泄露**，prompt 再怎么警告也只是降低概率，不能根除
3. **运行时 LLM 调用 = 一整条可能翻车的链路**：模型服务限流、网络抖动、JSON 解析失败、幻觉、自泄…… 每一项都可能让一局变差

**结论**：这些不是 prompt 调优能根治的问题，必须**把"出题"这件事从运行时挪到离线**。

## 2. 路线选择

三种题库化路线横向对比：

| 维度 | 路线 A<br>题库存完整题（1 套线索） | 路线 B<br>题库只存答案，线索 LLM 现编 | **路线 C<br>题库存答案 + N 套线索** |
|---|---|---|---|
| 运行时 LLM 调用 | 0 | 每题 1 次 | **0** |
| 单题延迟 | <100ms | 7-15s | **<100ms** |
| 熟脸答案再出的"线索新鲜感" | ❌ 完全一样 | ✅ 现编，每次不同 | ✅ N 套轮换 |
| self-leak 风险 | 无（生成期已审） | 高（每次现场随机） | 无（生成期已审） |
| 题库体积 | 最小 | 最小 | N 倍（可接受） |
| 生成期工作量 | 基准 | 无 | N 倍 |
| 成熟产品范例 | —— | —— | 头脑王者 / 一站到底等知识竞答 App |

### 2.1 为什么选 C

- **运行时零 LLM** 是题库化的核心收益，不能为"新鲜感"牺牲掉 → 排除 B
- **同答案再出线索一字不差** 对熟悉的群友极损体验 → 排除 A
- 生成期 N 倍成本是**一次性投入**，换来运行时永久无感 → C 的成本可接受

### 2.2 N 取多少

| N | 总线索条数 | 生成期 LLM 调用 | 同答案在同局出现两次的概率 | 玩家记住单套线索的概率 |
|---|---|---|---|---|
| 1（= 路线 A） | 580 | ~580 | 已由 avoid 机制禁止 | 每次完全一样 |
| **2** | 1160 | **~1200** | 跨局 | 两套轮换，较难记住 |
| 3 | 1740 | ~1800 | 跨局 | 三套基本记不住 |

**选 N=2**。核心考虑：

- 两套已足够：跨局第二次抽到同答案时，**不同的线索顺序/角度足以让玩家感觉是新题**
- 成本可控：LLM 调用从 ~600 翻到 ~1200（而不是 ~1800），生成期 1.5x 而非 3x
- 可扩展：如果后续发现 2 套不够，生成脚本可以再跑一轮**补第 3 套**（增量模式，不用重做）

## 3. 每类题库规模（带依据）

### 3.1 总则

- **重复控制公式**：想玩 K 局（10 题/局）不撞，题库应 ≥ 10K
- **覆盖广度 vs 玩家熟悉度**：再多的冷门答案也不是玩家想要的。规模 = min(LLM 产出上限, 玩家熟悉度上限, 重复控制需求)
- 群友典型玩耍频率：一周 3-5 局 / 类，两周内不撞答案即算合格

### 3.2 差异化配额

| 类型 | 题数 | 理论池上限 | 决策依据 |
|---|---|---|---|
| **country** | **80** | 全球 ~200 国，"广为人知"约 60-80 | 再多会出瑙鲁/斐济这种无特征小国，玩家猜不出 → 80 是池子的自然上限 |
| **city** | **120** | 中外知名城市 150+ | 中国 TOP 30 + 国际 TOP 80 + 一些特色城市（开罗/伊斯坦布尔）接近 120 |
| **food** | **120** | 全球菜品 150-250 | 中/西/日/韩/东南亚/甜品 主力品 120 左右，再多开始出冷门 |
| **person** | **100** | 真人 + 经典虚构合计 ~130 | 真人名人（科学家 20/艺术家 20/作家 20/企业家 10/运动员 20/ 其他 10）+ 虚构（神话 15/经典文学 15）≈ 110，留 10 缓冲 |
| **animal** | **80** | "大家认得" 的动物 80-120 | 再多就是袋熊/狞猫这种冷门，非爱好者直接懵 |
| **idiom** | **80** | 有典故成语 200-300 | **但"每个中国人都知道"的只有 60-80 个**（完璧归赵/刻舟求剑/画蛇添足这类）。80-150 那段（洛阳纸贵/秦晋之好）开始有人猜不出 |
| **合计** | **580** |  |  |

### 3.3 各类能支撑多少局不撞

- country 80 → 8 局（两周 OK）
- city 120 → 12 局（两周 OK）
- food 120 → 12 局（两周 OK）
- person 100 → 10 局（两周 OK）
- animal 80 → 8 局（两周 OK）
- idiom 80 → 8 局（两周 OK）

**两周内玩家不会遇到同答案**。跨两周后允许跨局重复（见 §6.3）。

## 4. 运行时架构

### 4.1 数据模型（每条题库项）

```json
{
  "answer": "爱因斯坦",
  "aliases": ["Einstein", "阿尔伯特·爱因斯坦"],
  "clue_sets": [
    [
      "他提出了一个改变了物理学的理论",
      "他在瑞士专利局任过职",
      "他的方程让时空变得可变",
      "他因光电效应获得了诺贝尔奖",
      "他那张吐舌头的照片广为流传"
    ],
    [
      "20 世纪最被仰慕的科学家之一",
      "他的理论提示了光速的不可超越",
      "他曾在普林斯顿高等研究院工作",
      "他的头发标志性地凌乱",
      "他和 E=mc² 这个公式紧紧绑定"
    ]
  ],
  "explanation": "20 世纪最伟大的物理学家之一，相对论的提出者。",
  "difficulty": "easy",
  "source": "llm_gen_2026-04-30_v1"
}
```

字段说明：
- `answer`：标准答案，用于宣告和匹配基准
- `aliases`：判定时的别名（原有宽松匹配沿用）
- `clue_sets`：**2 套线索**，每套严格 5 条，从隐晦到直白
- `explanation`：答对/跳过后展示，全题共用（不随 clue_set 变化）
- `difficulty`：`easy` / `medium`（70% easy + 30% medium，生成期标注）
- `source`：生成来源，便于追溯

### 4.2 文件组织

```
seeds/
  trivia_bank/
    country.json     # list[PuzzleEntry]
    city.json
    food.json
    person.json
    animal.json
    idiom.json
```

### 4.3 运行时出题流程

```python
def get_next_puzzle(type_id, avoid: list[str]) -> TriviaPuzzle:
    bank = load_bank(type_id)                         # 启动时缓存，O(1)
    # 跳过本局已出过的答案/别名
    candidates = [p for p in bank if not _is_avoided(p, avoid)]
    if not candidates:
        # §6.3 池空策略：允许跨局重复 或 本局结束
        candidates = bank
    entry = random.choice(candidates)
    clue_set = random.choice(entry["clue_sets"])      # 从 2 套里抽一套
    return TriviaPuzzle(
        type_id=type_id,
        answer=entry["answer"],
        aliases=entry["aliases"],
        clues=clue_set,                                # 一套 5 条
        explanation=entry["explanation"],
    )
```

**零 LLM 调用**。延迟取决于随机选中 + 查表，<100ms 稳定。

### 4.4 本局 avoid 机制沿用 v1.3

题库抽题时直接跳过 history 里已出过的答案/别名。无需改现有 `GameBase._collect_used_names` 逻辑。

## 5. 生成脚本设计

### 5.1 文件

`scripts/generate_trivia_bank.py`

### 5.2 命令行

```bash
# 生成 country 类，目标 80 道，每题 2 套线索
uv run python scripts/generate_trivia_bank.py \
    --type country --count 80 --clue-sets 2

# 断点续传（读取已生成的，补足到目标数）
uv run python scripts/generate_trivia_bank.py --type person --count 100 --resume

# 批量跑所有类型
uv run python scripts/generate_trivia_bank.py --all
```

### 5.3 生成流程（单类型）

```
1. 加载已有 seeds/trivia_bank/<type>.json（若存在） → existing_answers
2. 循环直到 len(bank) >= target_count:
   a. 从 existing_answers 取最近 30 个作为 avoid
   b. 调 trivia_host 生成 1 道候选题（复用现有 generate_puzzle）
   c. 自检（现有 _validate，avoid_norms 严格模式）
   d. 去重（normalize 后不能与 existing_answers 任一命中）
   e. 过关 → 生成第 2 套线索（另起 prompt 让 LLM 写同答案的不同角度线索）
   f. 第 2 套同样自检
   g. 两套都 OK → 写入 bank.json（原子写）
3. 打印进度 / 写日志
```

### 5.4 第 2 套线索生成 prompt（新增）

```
你将为同一个答案生成第二套线索，用于问答游戏的题库。

答案：{answer}
已有第 1 套线索（不要重复）：
1. ...
2. ...
...

请从**完全不同的角度**写 5 条新线索，遵循以下要求：
- 从难到易排列
- 不得出现答案或别名的任何字
- 角度要和第 1 套显著不同（第 1 套讲了成就，第 2 套可以讲生平；
  第 1 套讲了地理，第 2 套可以讲文化）
- 只输出 JSON：{"clues": [...]}
```

### 5.5 idiom 类的特殊处理

成语**典故池小**（~100 个），LLM 随机生成会大量撞车。流程：

```
1. 先调 LLM 要一份"有典故成语候选清单"（目标 100 个）
2. 人工快速过一遍清单，去掉重复/不合规的
3. 循环清单逐个调 LLM 生成完整题目（比随机生成高效 10 倍）
```

### 5.6 断点续传

- 每生成 1 道就 **原子写入** `seeds/trivia_bank/<type>.json`（写 tmp 再 rename）
- 重启时 `--resume` 读取已有文件，补足到 target_count

### 5.7 并发与限流

- 单线程串行（简单 + 避免智谱限流 429）
- 两次调用之间 sleep 0.2s

### 5.8 审核产出

生成结束后打印一份报告：

```
[country] 目标 80 道 · 实际 80 道 · LLM 调用 174 次 · 生成成功率 92%
[rejected] 15 道被拒
  - 5 条 self-leak
  - 7 条重复
  - 3 条字段不全
[写入] seeds/trivia_bank/country.json (80 items × 2 clue_sets)
```

被拒的题写入 `seeds/trivia_bank/country_rejected.json` 方便回看。

## 6. 其他关键决策

### 6.1 不保留 LLM 兜底

理由：
- 题库 580 道（每类 80-120）× 10 题/局 = 每类 **8-12 局不撞**，现实跑不光
- 兜底 = 一条额外代码分支 + 一套测试 + 运行时可能翻车的 LLM 调用，**收益远小于成本**
- 保持"纯题库"的定位干净

### 6.2 不要烂题淘汰

理由：
- 生成期代码自检 + 人工抽查已经把 self-leak 类问题挡住
- 问答题目事实正确性远好于海龟汤（海龟汤的"烂题"很多是 LLM 讲崩了故事，问答没这问题）
- 加淘汰机制 = 一张表 + 一个指令 + 复杂的 per-group 状态

### 6.3 题库跑光策略

**方案 A（推荐）：允许跨局重复抽取**。

理由：
- 跨局间隔至少一局（几分钟到几小时），玩家记忆早已淡化
- 加上 2 套线索随机抽取，熟脸答案也有新鲜感
- 实现极简：运行时不做任何跨局记录

**方案 B（备选）：本局按剩余题数缩短**。
- 如果本局 avoid 和题库碰撞，出不满 10 题就按实际数结算
- 一般不会触发（除非刻意连玩 8 局同类型）

**采用 A 作为默认**。`_is_avoided` 只作用于本局，跨局不记录。

### 6.4 难度标记（difficulty）

- `easy`：常识级（"中国首都"、"《西游记》的主角师父"）
- `medium`：需要一定知识（"法国大革命爆发年代相关的城市"）
- 配比 **70% easy + 30% medium**，通过生成 prompt 控制（让 LLM 自标）

运行时**当前不按难度筛选**（保持简单），但字段预留，后续可做"难度模式"。

## 8. 验收标准

### 8.1 功能

- [ ] 运行 10 局 `/开始 trivia country`，单局内 0 重复答案
- [ ] 连续跑 5 局同类型，跨局重复率 < 30%
- [ ] 单题从玩家"要下一条线索"到机器人响应 **< 300ms**（目前 LLM 模式约 7-15s）
- [ ] 整局（10 题）开始到结算 **< 30 秒**（纯玩家交互时间，不含线索阅读）

### 8.2 质量

- [ ] 每类随机抽 10% 的题，人工 review：
  - 答案/别名不在线索里（self-leak = 0）
  - 答案明确唯一（不会歧义）
  - 5 条线索从难到易有梯度
  - 解释文不超 60 字、有信息量
- [ ] 所有单元测试通过

### 8.3 文档

- [x] 本设计文档
- [ ] `docs/games/trivia.md` 更新到 v1.4
- [ ] `docs/10-roadmap.md` 登记
