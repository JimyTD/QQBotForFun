# Test Runs · 验收流水归档

这个目录存放**人工触发的验收测试流水文档**。和单元测试（`tests/`）是两回事：

| 类型 | 位置 | 用途 | 触发方式 |
|---|---|---|---|
| 单元测试 | `tests/` | 逻辑正确性 | `pytest` 自动跑，CI 必过 |
| **验收流水** | `docs/test-runs/` | **LLM 产出的调性/质量验证** | 手动跑脚本，人工阅读 |

LLM 行为没法在单元测试里断言（每次输出都不一样），但每次改 prompt 都需要看真实产出判断效果。所以把这些样本留档，便于对比演进。

## 命名规范

```
<YYYYMMDD>[-<HHMMSS>]-<主题>[-<子场景>].md
```

- `YYYYMMDD`：运行日期
- `HHMMSS`（可选）：同一天多次运行时自动加上，避免覆盖
- `主题`：被验证的对象，如 `turtle-soup-category-v2`（海龟汤 v2.0 分类策略）
- `子场景`（可选）：例如 `悬疑` 表示这是定向测试悬疑类的样本

示例：
- `20260429-turtle-soup-category-v2.md` — 4 月 29 日随机 8 局
- `20260429-194236-turtle-soup-category-v2-悬疑.md` — 同日定向 4 局悬疑

## 产出脚本

当前在用的产出脚本：

| 脚本 | 主题 | 说明 |
|---|---|---|
| `scripts/test_soup_variety.py` | `turtle-soup-category-v2` | 海龟汤出题调性验收；支持 `force_category` 定向 |

新增验收脚本时请在此表登记，并把输出目录固定为 `docs/test-runs/`。

## 保留策略

- **同主题至少保留最近 1 份**，方便与下一次改动对比
- **重大版本升级点**（比如 HOST Prompt v1.0 → v2.0 这种）保留该版本的首份流水作为历史锚点
- 中间调试产生的重复样本可以删，不要让目录无限膨胀
- 删除前先确认这份流水没在别处（PR 讨论、issue、其他文档）被引用

## 阅读流水时的关注点

流水文件通常包含：

1. **分布概览**：目标分类/难度的采样分布，以及 LLM 回写一致性
2. **每局详情**：每条样本的 category/difficulty/汤面/汤底/线索 + 耗时
3. **结论 checklist**：人工填写的定性判断（是否达标、是否有瑕疵）

看样本时别只看成功率，**重点看"调性是否对"**：
- 悬疑样本是否真的有反转感？还是平淡描述？
- 奇幻样本的汤底是不是"现实解释"？有没有偷懒给超自然结局？
- 温情样本有没有情感后劲？
- 难度 1-5 是不是真的有梯度？
